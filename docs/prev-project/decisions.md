# Журнал решений

> Здесь — **почему** проект устроен так, а не иначе. Каждое значимое
> решение: что выбрали, какие были варианты, почему именно это.
> Файл **только дописывается** (новые записи — вниз). Старые записи не
> удаляем: если решение отменили — добавляем новую запись «отменяет D-NNN».

> Зачем: чтобы следующий исполнитель не «переоткрывал» уже закрытые вопросы
> и не ломал то, что сделано осознанно. Один абзац здесь экономит часы.

Записывай решение, когда выбор был **неочевидным** или у него есть
последствия: скоупы, структура данных, плейсменты, обход бага и т.п.

## D-001 · Шаблон _b24-single-php (single-tenant PHP)
**Дата:** — · **Статус:** принято

**Контекст.** Стартовали новый проект для Битрикс24.
**Решение.** PHP-бэкенд, файловый store (`<?php exit;?>`-защита), vanilla JS, Netangels.
**Почему.** Готовый скелет с OAuth, install-flow, session-gate «из коробки».
Без БД, без composer — минимум зависимостей для single-tenant.
**Последствия.** Multi-tenant / маркет — смена шаблона, не доработка этого.

## D-002 · Переезд хостинга на VPS + Caddy + PHP-FPM
**Дата:** 2026-07-03 · **Статус:** отменяет хостинг-часть D-001

**Контекст.** D-001 фиксировал хостинг как Netangels (shared, `.htaccess`).
Команда перешла на собственный VPS с Caddy + PHP-FPM для всех B24-приложений.

**Варианты.** Остаться на shared-хостинге (Netangels) vs собственный VPS с
явной настройкой каждого проекта через Caddyfile.

**Решение.** VPS, домен `b24.blackboxbegin.space`, один Caddy-блок `handle
/<slug>*` на проект, деплой — локальный rsync (`bash deploy.sh`) прямо на
сервере, без SSH.

**Почему.** Больше контроля над PHP-FPM/Caddy-конфигом, не завязаны на
ограничения конкретного shared-хостера. Caddy не читает `.htaccess` —
защита state осталась той же (`<?php exit;?>`-префикс), это не пришлось
менять.

**Последствия.** `.htaccess`-независимость шаблона (D-001) оказалась кстати:
переезд не потребовал менять модель защиты. Появились новые следствия
Caddy `uri strip_prefix` — APP_URL нужно править вручную после `init.php`,
handler URL обязательно с `/index.php`, `template.html` вместо `index.html`.
Все зафиксированы в `CLAUDE.md` → «Caddy + PHP-FPM — следствия для шаблона».
Домен, а не IP — при смене хостера под тем же доменом менять ничего не
нужно; деплой на сервер заказчика с другим доменом — отдельный кейс,
не покрыт этим решением.

## D-003 · env.php генерируется deploy.sh, init.php — только для 1%-кейса
**Дата:** 2026-07-04 · **Статус:** принято, уточняет D-002

**Контекст.** После D-002 конвент деплоя зафиксирован: домен всегда
`b24.blackboxbegin.space`, путь всегда `/var/www/b24/<slug>`. При этом
`APP_URL`/`APP_PATH`/`DATA_ROOT` стали полностью детерминированы из
одного значения (slug), но флоу продолжал требовать открыть `init.php`
в браузере и вручную поправить `APP_URL` (следствие `uri strip_prefix`
из D-002). На практике оба реальных проекта на VPS (`ap-simple-test`,
`fv-taskindep-panel`) этот шаг обошли — `env.php` был написан вручную
напрямую, `init.php` на сервере не оказывалось вообще (`deploy.sh` его
и не копирует, см. `--exclude 'init.php'`), задокументированный флоу
не воспроизводим на свежем сервере как описано.

**Варианты.** (а) Почистить `deploy.sh`, чтобы `init.php` таки доезжал
до сервера при первом деплое — оставить flow как есть. (б) Убрать
`init.php` из основного flow, генерировать `env.php` детерминированно
прямо в `deploy.sh`, оставить `init.php` в шаблоне только как fallback
для хостинга, где путь/домен заранее не известны (shared-хостинг
заказчика, FTP).

**Решение.** (б). `deploy.sh` при первом деплое slug'а (env.php на
сервере ещё нет) генерирует `env.php` из `env.example` подстановкой
`APP_URL=https://b24.blackboxbegin.space/<slug>`, `APP_PATH=DEPLOY_DIR`,
`DATA_ROOT=DEPLOY_DIR/data`. Человеку остаётся вписать только
`B24_CLIENT_ID`/`SECRET` — их и так нельзя получить иначе, кроме как
из карточки local-app в Б24. `init.php` физически остаётся в шаблоне
(`www/init.php`) как есть, без изменений — для кейса, где конвент
VPS+Caddy не применим (хостинг заказчика с непредсказуемым путём).

**Почему.** Автоопределение `APP_URL` через `init.php` было нужно для
общего случая (произвольный shared-хостинг, куда шаблон изначально
целился) — на своём VPS с фиксированным конвентом это лишний шаг,
который к тому же не работал как описано (см. «Контекст»). Убирать
`init.php` из репозитория целиком не стали — шаблон должен оставаться
пригодным не только для нашего VPS.

**Последствия.** `how-to-link.md` описывает только 99%-кейс (наш VPS,
`deploy.sh`). Новый файл `how-to-link-ALL.md` — процедура для 1%-кейса
через `init.php`, на него ссылаются `how-to-link.md`, `README.md`,
`CLAUDE.md`. `docs/state.md`-чеклист (шаблонный, наследуется новыми
проектами) обновлён под новый flow.

## D-004 · Макет пульта — вариант B (таблица + KPI-сводка), без кэша на первой итерации
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Нужно было выбрать макет главной страницы пульта и способ
загрузки данных смарт-процессов (Сделка/Этап/Модуль/Оплата, ~4000 модулей
на портале). Сделано три черновика: A — плоская таблица без сводки,
B — таблица + KPI-плашки сверху, C — группировка по стадии.

**Варианты.** Макет: A / B / C (см. выше). Загрузка данных: считать
агрегаты (счётчики по стадиям, индикаторы срыва/оплаты) на клиенте после
раскрытия строки, либо посчитать их на сервере сразу и отдать фронту
готовое дерево. Кэширование: заводить файловый кэш сразу или отложить.

**Решение.** Макет — B (KPI-плашки сверху + таблица сделок + аккордеон
Сделка→Этап→Модуль). Данные — один эндпоинт `www/api/dashboard.php`,
собирает всё через `batch()`-запросы (`crm.item.list` постранично) и
агрегирует на сервере в `fetchDashboardData()` (`api/lib.php`); раскрытие
строки на фронте — чистый рендер уже полученного дерева, без новых REST.
Кэш — не делаем на этой итерации.

**Почему.** B даёт директору сводную оценку (сколько сделок, на какую
сумму, сколько горит) без скролла таблицы — то, ради чего вообще
затевался пульт, а не просто список сделок (A) или разбивка не по тому
измерению, которое важно директору (C, по стадии). Агрегация на сервере
и единый батч-запрос при открытии страницы — раскрытие аккордеона
мгновенное и не плодит REST-вызовы (см.
`rules/rule-b24-rest-batch-not-loop.md`); ожидаемый объём (~4000 модулей)
укладывается в 2-3 batch-запроса. Кэш отложен — на первой итерации важнее
корректность агрегатов, а не задержка одного REST-похода при открытии
страницы; преждевременная оптимизация.

**Последствия.** `fetchDashboardData()` в `api/lib.php` — точка, куда
позже можно добавить файловый кэш с TTL, не трогая остальной код и
фронт. UI и парсинг ответа `dashboard.php` пишутся под контракт «дерево
сделка→этапы→модули с готовыми агрегатами», а не «плоские списки
сущностей» — см. `docs/superpowers/specs/2026-07-13-pult-rukovoditelya-design.md`.

## D-005 · Деплой сразу на VPS rub24, минуя b24.blackboxbegin.space
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Начали разворачивать `ap-pdir` по конвенту шаблона (VPS
`b24.blackboxbegin.space`): задеплоили `bash deploy.sh`, зарегистрировали
local-app в Б24 с handler'ом на этот домен, получили `client_id`/`secret`.
Install-flow не завершился (Caddyfile на этом сервере не был настроен под
`/ap-pdir`, а правка требует root — `deploy`-юзер не может писать в
`/etc/caddy/`). До того как чинить это на старом сервере, пользователь
попросил остановиться: проект переезжает на другой VPS — по образцу
`ap-pp`, который уже прошёл аналогичный переезд на `rub24.blackboxbegin.space`
(`45.91.55.178`, см. D-009 в `ap-pp/docs/decisions.md`) ради меньшей
задержки до портала `alfa-prj.bitrix24.ru`.

**Варианты.** (а) Починить Caddyfile на `b24.blackboxbegin.space` (нужен
root/sudo с паролем — блокер) и остаться на нём. (б) Сразу деплоить на
`rub24.blackboxbegin.space`, повторив паттерн `ap-pp/deploy.sh` (SSH,
ключ `id_ed25519_rub24`, `deploy`-юзер там владеет `/etc/caddy/Caddyfile`
напрямую — правка без sudo).

**Решение.** (б). `ap-pdir` ещё не был живым (ни одной успешной установки,
токены не сохранились) — в отличие от `ap-pp` это не «cutover» с живого
прод-инстанса, а выбор целевого сервера с нуля. `deploy.sh` переписан по
образцу `ap-pp/deploy.sh` (SSH вместо локального rsync). Local-app в Б24
переиспользован — тот же `client_id`/`secret`, но handler/install path
пере-регистрируются на новый домен.

**Почему.** У `rub24` ниже сетевая задержка до портала (см. измерения в
D-009 проекта `ap-pp`: 77мс вместо 401мс round-trip) и `/etc/caddy/Caddyfile`
там правится без root — меньше трения при добавлении новых проектов.
Раз `ap-pdir` не запущен в проде, нет причины сперва разворачивать на
`b24.blackboxbegin.space` и потом переезжать — дешевле сразу выбрать
целевой сервер.

**Последствия.** `deploy.sh`, `CLAUDE.md` (`Деплой`, `О ПРОЕКТЕ`,
`ИНФРАСТРУКТУРА СЕРВЕРА`) и `how-to-link.md` переписаны под rub24.
Остаток проектов на старом сервере (`ap-simple-test`, `fv-taskindep-panel`,
`ap-kilotest2`) не затронут — это решение только про `ap-pdir`, не про
перенос всего домена (см. вариант (в), отклонённый в D-009 `ap-pp` по той
же причине). Локальный артефакт `/var/www/b24/ap-pdir` (создан этой
сессией на исходном VPS, где крутится сам Клод) удалён — недостижим,
Caddyfile под него так и не был написан.

## D-006 · Фильтр сделок (active/all/closed) — постфильтр в PHP, не REST-filter; фетч Pay добавлен в архитектуру
**Дата:** 2026-07-13 · **Статус:** принято, уточняет D-004

**Контекст.** При реализации `fetchDashboardData()` (см. спеку и D-004)
обнаружились два расхождения с архитектурой из спеки:
1. Спека определяет 🟡-индикатор через стадию записи **Pay** («Счёт
   отправлен», entityTypeId 1058), но раздел «Архитектура загрузки данных»
   перечисляет только 3 REST-шага (1050/1054/1062) — Pay не упомянут.
2. Реальный REST-ответ `crm.item.list` для `select`/полей — camelCase
   (`stageId`, `ufCrm13OCode`), а не классические коды из `/source`
   (`STAGE_ID`, `UF_CRM_13_O_CODE`) — см. новое правило
   `rules/rule-crm-item-camelcase-select.md`. Дальше вопрос: фильтровать
   пресет «Активные/Все/Завершённые» через REST `filter` по `stageId`
   (нужен полный код вида `DT1050_21:NEW` — категория не гарантирована
   без отдельного запроса) или в PHP после фетча.

**Варианты.** (1) Pay: добавить 4-й REST-шаг в архитектуру или убрать
Pay-условие из индикатора. (2) Фильтр пресета: REST `filter` с полным
префиксом стадии (`DT<entityTypeId>_<categoryId>:CODE`) — требует заранее
знать `categoryId` — или пост-фильтр в PHP по суффиксу `stageId` после
`:` (не требует знания `categoryId`, дешёво при реалистичном объёме
~50 сделок).

**Решение.** (1) Добавлен 4-й шаг: `crm.item.list` entityTypeId=1058,
`filter: {parentId1054: [...]}` по id уже загруженных этапов —
выполняется только если этапы вообще нашлись. (2) Пресет фильтруется в
PHP: сделки грузятся из Б24 без REST-фильтра по стадии, `stageCode`
(суффикс после `:`) сравнивается со списком закрытых стадий
(`SUCCESS`/`FAIL`) в `dashboardDealMatchesPreset()`.

**Почему.** (1) Без Pay 🟡-индикатор «Ожидание оплаты» неполный — часть
случаев (счёт выставлен, но этап ещё не в стадии «Оплата») не ловится.
Раз спека явно описывает это правило в разделе «Правила статуса», это
пробел в разделе «Архитектура», а не сознательный вырез — дозаполнено.
(2) Постфильтр в PHP не зависит от `categoryId` (который пришлось бы
получать отдельным запросом `crm.category.list` или хардкодить, рискуя
разойтись при смене категории на портале) и на объёме ~50 сделок не
создаёт заметных накладных расходов — REST-оптимизация по фильтру из
спеки была рассчитана на кейс с большим числом сделок, которого пока нет.

**Проверено.** Реальным REST-вызовом на портале `alfa-prj.bitrix24.ru`:
6 сделок, ручной пересчёт `lagDays`/индикаторов/KPI по сырым данным
Milestone/Module совпал с выводом `fetchDashboardData()` для всех трёх
пресетов (`active`/`all`/`closed`).

**Последствия.** Если объём сделок вырастет на порядок, постфильтр в PHP
стоит пересмотреть в пользу REST-`filter` (уже зная `categoryId` из
реального использования). `www/api/dashboard-data.php` — бизнес-логика
вынесена туда отдельно от generic `www/api/lib.php` (см. комментарий в
шапке `lib.php`), эндпоинт `www/api/dashboard.php` — тонкая обвязка
(session-gate + JSON).

## D-007 · Цвета стадий — сняты с воронки Б24 через crm.status.list, не подобраны на глаз
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** После первой проверки UI в браузере пользователь показал
скриншоты воронок Deal/Milestone/Module в настройках Б24 (у каждой
стадии — свой цвет, заданный админом) и попросил применить те же цвета
к бейджам стадий на пульте, а не изобретать свою палитру.

**Решение.** Цвета не сняты «на глаз» со скриншота — получены точным
REST-вызовом `crm.status.list(filter: {ENTITY_ID: "DYNAMIC_<entityTypeId>_STAGE_<categoryId>"})`
для каждой из 4 сущностей (categoryId подтверждён отдельно через
`crm.category.list`: 1050→21, 1054→23, 1062→27, 1058→25) и захардкожены
в `DASHBOARD_DEAL_STAGES`/`DASHBOARD_MILESTONE_STAGES`/`DASHBOARD_MODULE_STAGES`
(`www/api/dashboard-data.php`) рядом с названием и порядком стадии.
Эндпоинт отдаёт `stageColor` вместе с `stageName`, фронт красит бейдж
через inline `style` и сам выбирает чёрный/белый текст по формуле
относительной яркости (`0.299r+0.587g+0.114b`, порог 0.6) — так же,
как визуально решает Б24 (проверено на всех 21 стадии из трёх воронок:
совпадает).

**Почему.** REST — источник истины: цвет админ мог поменять в любой
момент, глазами со скриншота легко ошибиться на пару тонов и разойтись
с реальной воронкой. Формула контраста, а не хардкод «эта стадия светлая,
эта тёмная» — не пришлось бы держать в уме и обновлять список при
добавлении новой стадии с любым цветом.

**Последствия.** Если админ перекрасит стадию в Б24, бейдж на пульте
не обновится сам — цвета захардкожены в PHP-константах (тот же trade-off,
что и у названий стадий, зафиксированных в D-004/D-006). Если это станет
проблемой — можно один раз при старте `fetchDashboardData()` подтягивать
`crm.status.list` вместо констант, ценой лишнего REST-вызова на каждое
открытие пульта. Заодно исправлен смежный CSS-баг, найденный на том же
скриншоте: `.error-banner { display: flex }` перебивал атрибут `hidden`
(одинаковая специфичность, автор-стиль позже UA-стиля в каскаде) — баннер
ошибки был виден пустым поверх успешно загруженных данных; добавлено
`.error-banner[hidden] { display: none }`.

## D-008 · Визуальный язык пульта — избранные элементы bugster.dev, без бренда/маскотов
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Пользователь показал referens-дизайн bugster.dev (лендинг
QA-инструмента, стиль «dev-tool стикербук»: зернистая бумажная текстура,
жирные чёрные обводки карточек с крупным скруглением, моноширинный шрифт
даже в заголовках/кнопках, мелкие квадратные «хендлы» по углам рамок как
в Figma, пастельные пилюли-кнопки, рукописные маскоты-иллюстрации) и
попросил применить этот стиль к пульту руководителя.

**Варианты.** (а) Буквально перенести весь визуальный язык, включая
пастельные заливки кнопок и иллюстрации-маскоты. (б) Взять только
структурные/графические приёмы (текстура, обводки, моношрифт, хендлы),
не трогая цветовую семантику пульта и не добавляя декоративных
иллюстраций. (в) Не менять дизайн вовсе.

**Решение.** (б). Перенесены: зернистая SVG-текстура фона (`feTurbulence`
data-URI) вместо прежнего точечного узора; чёрные обводки (`--ink`) и
увеличенный радиус скругления у KPI-блока и таблицы; KPI-плашки собраны
в одну рамку с разделителями вместо 4 отдельных карточек (паттерн
«10x / <5min / Zero» с исходника); моноширинный шрифт (`--font-mono`) —
заголовок, KPI-числа, шапка таблицы, кнопки-пресеты; квадратные
`.corner-handle` по углам обеих рамок. Не перенесено: пастельная палитра
кнопок/заливок, маскоты и декоративные иллюстрации, полный переход на
моноширинный шрифт (плотный текст ячеек — названия, активность —
остался sans-serif). Цвета плашек стадий (D-007, реальные цвета из
воронки Б24) не тронуты ни на йоту — это не брендинг, а данные.

**Почему.** Пульт — рабочий инструмент для директора и замов, не
маркетинговый лендинг: маскоты и пастельные стикеры создали бы
несерьёзное впечатление и отвлекали бы от чисел. При этом графические
приёмы (текстура вместо плоского белого, жирная обводка вместо мягкой
тени, моношрифт для чисел/лейблов) — стилистически нейтральны и хорошо
ложатся на «control-panel»-природу самого приложения (см. название
«Пульт руководителя»). Кириллица в моноширинном шрифте в длинных строках
(названия сделок, тексты активности) читается хуже, чем в sans-serif —
поэтому моно ограничен короткими элементами (числа, лейблы, код).

**Последствия.** Дальнейшие UI-доработки пульта следует держаться этого
языка (текстура/обводки/моно для чисел и лейблов), а не возвращаться к
плоским белым карточкам с мягкими тенями — если понадобится другое
направление, это отдельное решение, не тихий откат. Цвета стадий и их
источник (`crm.status.list`, D-007) — по-прежнему единственный источник
цветовой семантики, бренд-референсы вроде bugster.dev на них не влияют.

## D-009 · Переход на сущность — иконка-ссылка + BX24.getDomain(), без правок бэкенда
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Пользователь попросил кнопки перехода со страницы пульта
на карточку сущности (Сделка/Этап-Milestone/Модуль) в Б24 — открывать
в новой вкладке. Нужно было решить: откуда брать домен портала для URL,
и как оформить переход, не ломая клик-обработчик разворачивания строк
аккордеона.

**Варианты.** Домен: (а) прокинуть через бэкенд (`b24.domain()` уже
доступен в `session.php`/`index.php`) в `window.APP_*`-инъекцию; (б) взять
на фронте динамически через `BX24.getDomain()` при старте, раз приложение
и так грузит SDK `BX24.init()`. Оформление: иконка-стрелка рядом с
названием / отдельная колонка «Открыть» / кликабельный код-номер.

**Решение.** (б) для домена — `BX24.getDomain()` вызывается в `boot()`
сразу после `BX24.init`, результат кладётся в `state.domain`, URL
строится по паттерну `crm/type/{entityTypeId}/details/{id}/`
(entityTypeId захардкожен в `ENTITY_TYPE_ID` — 1050/1054/1062, как и
остальные ID сущностей в проекте). Оформление — иконка ↗ сразу после
названия во всех трёх уровнях (сделка, этап, модуль). Обработчик клика
в `tbody` явно игнорирует `.entity-link` (`e.target.closest('.entity-link')`),
чтобы клик по иконке не триггерил заодно разворачивание/сворачивание
строки.

**Почему.** `BX24.getDomain()` не требует правок бэкенда/эндпоинта
данных — id и entityTypeId уже были в дереве, которое отдаёт
`dashboard.php`; домен нужен только на фронте и только для построения
href. Иконка рядом с названием не занимает отдельную колонку в и так
плотной таблице и не путается с остальными кликабельными элементами
строки.

**Последствия.** Ссылка работает только внутри B24-iframe (там, где
`BX24.getDomain()` доступен) — при отсутствии `BX24` (гипотетический
прямой заход мимо портала) иконка просто не рендерится, это уже
покрыто существующей веткой `if (window.BX24) BX24.init(boot); else boot();`.
Если понадобится открывать сущность в контексте прямого URL другого
портала (multi-tenant) — потребуется пересмотреть источник домена,
не покрыто этим решением (см. `CLAUDE.md`: multi-tenant — смена
шаблона).

## D-010 · KPI-плашки «срыв графика»/«ожидают оплаты» — счётчик этапов не гейтится ранней стадией сделки
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Пользователь попросил показывать в скобках рядом с числом
сделок суммарное число этапов (Milestone), удовлетворяющих тому же
условию — «2 (4)», «1 (1)» — чтобы видеть картину не раскрывая
аккордеон. Существующий подсчёт индикаторов на уровне сделки исключает
сделки в ранних стадиях («Подписание», «Авансирование») — `isEarly`-гейт
в `fetchDashboardData()`. Вопрос: должна ли цифра-в-скобках (по этапам)
применять тот же гейт, или считать этапы всех незавершённых сделок
независимо от стадии самой сделки.

**Варианты.** (а) Тот же гейт — скобка логически согласована с основным
числом (все просроченные этапы принадлежат тем же сделкам, что вошли в
«2»). (б) Без гейта — независимый общий счётчик просроченных/ожидающих
оплаты этапов по всем незавершённым сделкам, включая ранние.

**Решение.** (б). Явный ответ пользователя: «Счетчик просрочки должен
смотреть по всем этапам сделок, которые еще не завершены. Чтобы
руководитель знал, сколько этапов у него будет не вовремя выполнено и
по скольким этапам он получит деньги позже, чем по договору». Счётчики
`brokenScheduleMilestoneCount`/`awaitingPaymentMilestoneCount` в `$kpi`
растут по каждому незавершённому этапу (`$mOpen`) во всех сделках
текущего пресета — без оглядки на `isEarly` и без оглядки на то, попала
ли сама сделка в число «2»/«1».

**Почему.** Директору важно видеть *весь* риск по этапам, включая те,
что относятся к формально ранним сделкам (подписание/авансирование) —
просроченный этап там не менее реален, просто сама сделка ещё не считается
«активной» по другому бизнес-критерию. Скрывать такие этапы из скобки
значило бы давать заведомо заниженную цифру.

**Последствия.** Число в скобках может быть **не согласовано** с суммой
видимых в таблице бейджей срыва по раскрытым сделкам (в раннем этапе)
— это осознанный trade-off, не баг. Проверено фикстурой, воспроизводящей
сценарий со скриншота пользователя (2 сделки/4 этапа-срыв, 1 сделка/1
этап-оплата) — `www/api/dashboard-data.php`.

## D-011 · Заказчик/Ответственный + счётчики стадий — «CRM-привязка» бывает без префикса, формат счётчиков сменён на упорядоченный список
**Дата:** 2026-07-14 · **Статус:** принято

**Контекст.** Реализовали колонки/фильтры «Заказчик»/«Ответственный»
(поля `UF_CRM_13_CUSTOMER_COMP`/`ASSIGNED_BY_ID` на Deal, см. спеку
`docs/superpowers/specs/2026-07-14-zakazchik-otvetstvenny-filtr-design.md`)
и заменили текстовые лейблы счётчиков этапов/модулей («2 заверш ·
1 ожид») на цветные плашки-числа по фидбеку пользователя (разъезд
текста, жёлтый плохо отличим от фона). Два отдельных, но связанных
неочевидных выбора внутри одной итерации:

1. Официальная документация Б24 (`data-types.md`, раздел «Формат
   значений для пользовательского поля „Привязка к элементам CRM“»)
   описывает формат `{PREFIX}_{ID}` (например `CO_123` для компании) —
   спека была написана под это. Проверка реальным REST-вызовом на
   `alfa-prj.bitrix24.ru` (`crm.item.fields`) показала, что
   `ufCrm13CustomerComp` имеет `settings: {"COMPANY":"Y","LEAD":null}` —
   поле ограничено ровно одним разрешённым типом CRM-сущности — и
   реальное значение в `crm.item.list` приходит **без префикса**,
   голым числом (`"5"`, а не `"CO_5"`). Парсер, написанный строго под
   документацию, резолвил `customerName` в `null` для всех сделок.
2. Порядок и цвет плашек этапов/модулей нужно было брать из
   существующих констант `DASHBOARD_MILESTONE_STAGES`/
   `DASHBOARD_MODULE_STAGES` — вопрос был, менять ли сам контракт
   `dashboard.php` (`milestoneCounts`/`moduleCounts`), ломая формат,
   которым уже пользуется фронт, или пытаться сохранить обратную
   совместимость.

**Варианты.** (1) Резолв CRM-привязки: (а) считать формат без
префикса ошибкой/edge-case'ом и требовать префикс всегда, чиня только
для конкретно этого поля отдельным путём; (б) обобщить парсер —
принимать оба формата (голый ID и `{PREFIX}_{ID}`), не завязываясь на
то, что поле обязано быть многотипным. (2) Формат счётчиков: (а)
оставить `{label: count}`, досортировать на фронте JS-объектом с
Map-подобным порядком (ненадёжно — порядок ключей объекта в JS не
гарантирован спецификацией для нечисловых строк, хоть и стабилен на
практике); (б) сменить сам контракт на `list<{stageCode,name,color,count}>`,
сортировку и цвет отдать бэкенду, который и так владеет константами
стадий.

**Решение.** (1)(б) — `dashboardParseCrmBindingId()` сначала проверяет
`ctype_digit($value)` (голый ID) и только если нет — ищет `{PREFIX}_{ID}`
через `strrpos('_')`. Оба пути взаимоисключающие (голый ID не может
содержать `_`), коллизий нет. (2)(б) — новая функция
`dashboardOrderedStageCounts(array $countsByStageCode, array $stageDefs): array`
строит список в порядке объявления констант стадий, с готовыми
`name`/`color`; `www/api/dashboard-data.php` меняет
`milestoneCounts`/`moduleCounts` на новый формат, фронт (`countBadges()`
в `www/js/app.js`) рендерит цветные плашки с числом и `title`-подсказкой
вместо текстовых лейблов.

**Почему.** (1) REST — источник истины (тот же принцип, что и в D-007
про цвета стадий): документация Б24 описывает общий случай, но не
покрывает частный случай поля, ограниченного одним типом сущности —
без проверки реальным вызовом баг остался бы незамеченным до продакшена
(разработка велась в git worktree, без прямого доступа к токенам
портала с этой машины — верификация шла через `sudo -u www-data php`
на сервере поверх уже задеплоенного `env.php`, без затрагивания живого
`api.dashboard.php`). (2) Явный контракт с сервера (порядок+цвет)
надёжнее, чем полагаться на порядок ключей JS-объекта, и не дублирует
знание о порядке стадий на фронте и бэкенде — единственный источник
истины остаётся в PHP-константах, как и раньше для названий/цветов.

**Последствия.** `dashboardParseCrmBindingId()` теперь общая для любых
CRM-полей типа `crm` независимо от того, ограничены они одним типом
сущности или нет — если в проекте появится многотипное поле, парсер уже
готов. `dashboard.php` — breaking change контракта `milestoneCounts`/
`moduleCounts` (было `{label: count}`, стало
`list<{stageCode,name,color,count}>`); нет внешних потребителей JSON
кроме собственного `www/js/app.js`, обновлённого в той же итерации —
совместимость держать не пришлось. `DASHBOARD_MILESTONE_SHORT_LABELS`/
`DASHBOARD_MODULE_SHORT_LABELS` оставлены в файле объявленными, но
неиспользуемыми (см. `rules/rule-crm-item-camelcase-select.md` по духу —
не удалять данные «на всякий случай», раз они не мешают). Проверено
реальным REST на `alfa-prj.bitrix24.ru` до и после фикса (`customer=NULL`
→ `customer=ФудМаркет ООО`), плюс визуально пользователем в браузере.

## D-012 · Индикаторы эмодзи (замена, не добавление) + выравнивание колонок только внутри своего уровня аккордеона
**Дата:** 2026-07-14 · **Статус:** принято

**Контекст.** Два связанных UI-полиша по фидбеку пользователя после
проверки предыдущей фичи (скриншоты 2026-07-14, см. `docs/state.md`):
(1) цветные точки 🔴🟡 в колонке «Индикаторы» мелкие, жёлтая плохо
отличима от зернистой текстуры фона таблицы; (2) при раскрытии модулей
у разных этапов колонка «Стадия» (и другие) под-таблиц скачет по
горизонтали между блоками — стадии этапов визуально путаются со
стадиями модулей.

**Варианты.** (1) Индикаторы: (а) заменить точки эмодзи целиком; (б)
оставить точку и добавить эмодзи рядом (точка = цвет-категория, эмодзи
= доп. пояснение). (2) Выравнивание: (а) выравнивать только внутри
своего уровня — все под-таблицы Этапов между собой одной ширины
колонок, все под-таблицы Модулей между собой одной ширины, но между
уровнями ширина может отличаться (отступы `.detail-wrap`/`.module-wrap`
и так намеренно разные, D-008); (б) свести колонку «Стадия» этапов и
модулей в одну вертикальную линию, для чего пришлось бы урезать/убрать
отступ вложенности; (в) единая сетка колонок Номер/Название/Стадия на
всю страницу (сделки+этапы+модули), самый инвазивный вариант.

**Решение.** (1)(а) — `indicatorDots()` в `www/js/app.js` рендерит
`⏰`/`💰` вместо `<span class="indicator-dot ...">`, `.indicator-dot`-CSS
удалён как более не используемый. (2)(а) — модификаторы класса
`sub-table--milestones`/`sub-table--modules` на `<table class="sub-table">`
в `www/js/app.js` (`dealMilestonesHtml()`/`milestoneModulesHtml()`) +
`table-layout: fixed` с процентными ширинами колонок в
`www/css/style.css`, раздельно для двух форм таблицы (6 колонок у
Этапов, 5 у Модулей — разный состав, поэтому не общий набор правил).

**Почему.** (1) Эмодзи самодостаточны по смыслу (⏰=срыв графика,
💰=оплата) — добавлять цвет для различения избыточно, снимает жалобу
на плохую контрастность жёлтого без замены на другой цвет. (2)
Отступ по глубине вложенности — часть визуального языка пульта (D-008),
жертвовать им ради единой линии means терять сигнал «это дочерний
уровень»; сведение в одну сетку на всю страницу (в) избыточно для
жалобы, которая была конкретно про «скачет между блоками одного
уровня», а не про кросс-уровневое совпадение. `table-layout: fixed` —
стандартный способ сделать ширину колонок HTML-таблицы независимой от
содержимого конкретного экземпляра, без переписывания на CSS Grid.

**Последствия.** Длинные тексты (названия этапов/модулей, тексты
активности), не помещающиеся в фиксированную ширину колонки, теперь
переносятся на несколько строк вместо того, чтобы раздвигать колонку —
ожидаемое поведение `table-layout: fixed`, не баг. Если понадобится
кросс-уровневое совпадение «Стадии» этапов и модулей — это отдельное
решение, требующее пересмотра отступов вложенности (вариант (б) выше),
не покрыто этим решением. Проверено пользователем глазами в Б24 после
деплоя (оба пункта).

## D-013 · Кнопка 🎯 «Создать задачу» — своя модалка, привязка всегда к Сделке, CRM-биндинг смарт-процессов через T{hex}_, постановщик через userId в сессии

**Решение.** (1) `UF_CRM_TASK` в `tasks.task.add` всегда строится от
`entityDealId`, независимо от уровня клика (`www/api/task-create.php`).
Формат привязки для смарт-процессов — НЕ `D_{id}` (это только для
классической Сделки `entityTypeId=2`), а `T{hex(entityTypeId)}_{id}`:
наша Сделка — смарт-процесс `entityTypeId=1050`, `dechex(1050)='41a'` →
`T41a_{id}` (сверено по
`/home/deploy/refs/b24-rest-docs/api-reference/crm/data-types.md#crm-binding-format`).
(2) Модалка на чистом JS/HTML/CSS (`www/template.html`, `www/css/style.css`,
`www/js/app.js`) — название/описание/исполнитель/срок, описание
предзаполняется шаблоном по уровню (`Задача к сделке/этапу/модулю «…»`),
редактируется вручную. (3) `www/api/session.php`: `createSession()`
получил `?string $userId = null`, резолвится через уже существовавший
`resolveUserIdFromAuth()` при создании сессии в `tryCreateSessionFromB24Post()`
— `task-create.php` берёт `CREATED_BY` из `$session['userId']`, и явно
отказывает (500), если он не резолвился, вместо тихого создания задачи
с неверным постановщиком. (4) Задача дополнительно помечается тегом
`Задача из пульта` (`TAGS` в `tasks.task.add`) — чтобы отличать от
автоматических задач БП модуля и задач, заведённых вручную внутри
самой карточки модуля. (5) Код поля-источника «Объект наименование»
(`UF_AUTO_824402759720` на задаче тянется из «Краткого наименования»
Сделки) — в `source/АП Deal.md` и `source/АП Задачи.md` были два разных
варианта кода; сверено живым REST-запросом `crm.item.fields` на
`alfa-prj.bitrix24.ru`: верный код — `UF_CRM_13_O_SNAME` (camelCase
`ufCrm13OSname`).

**Почему.** Смарт-процессы Б24 — не то же самое, что классические CRM-сущности:
их id-префикс в UF-полях типа «привязка к CRM» вычисляется из
`entityTypeId` через hex, а не берётся из фиксированной таблицы
кратких кодов (`D`/`C`/`CO`/…) — тот же класс граблей, что и в
D-011 (не полагаться на общую доку без проверки на конкретном
портале/типе сущности). Явный `CREATED_BY` из сессии — прямое
следствие архитектуры single-tenant local-app с одним общим токеном
(см. `CLAUDE.md`): без этого «постановщик» был бы технически неверным
полем в каждой созданной через пульт задаче.

**Последствия.** После сохранения нового scope `task` в карточке
local-app уже выданный `access_token` не подхватывает его ретроактивно
— потребовалось форсировать обновление через `grant_type=refresh_token`
(одноразовый ручной шаг при первом включении фичи на проде, не часть
кода приложения — задокументировать в `how-to-link.md`/README, если
грабля повторится на будущих scope-расширениях). Если в будущем
понадобится привязка к Этапу/Модулю отдельно от Сделки — это отменяет
пункт (1) этого решения, потребует пересмотра.

## D-014 · Срыв срока на уровне Модуля — считаем в приложении по полям-датам стадий, не по автоматизации БП
**Дата:** 2026-07-15 · **Статус:** принято

**Контекст.** На уровне Этапа (Milestone) срыв срока уже считался готовым
полем `UF_CRM_15_MST_CONTR_PLAN` — бизнес-процесс сам вычисляет лаг в
днях. На уровне Модуля (1062) такой автоматизации в БП нет и нет
соответствующего готового поля — пользователь явно попросил закрыть
этот пробел на уровне приложения. См. спеку
[docs/superpowers/specs/2026-07-15-srok-modulya-design.md](superpowers/specs/2026-07-15-srok-modulya-design.md)
и план
[docs/superpowers/plans/2026-07-15-srok-modulya.md](superpowers/plans/2026-07-15-srok-modulya.md).
Реализовано через subagent-driven-development на ветке
`worktree-srok-modulya`, 3 задачи + фиксы по итогам финального ревью,
все прошли implementer+reviewer цикл, ветка слита в `master` и
задеплоена.

**Варианты.** (1) Источник «даты завершения текущей стадии» модуля:
завести новое поле в БП/CRM, или переиспользовать уже существующие
поля-даты по стадиям (`UF_CRM_19_MOD_RUN`/`_CHECK`/`_CREATE`/`_EDIT`/
`_WAIT`/`_APPROVE`), названия которых 1:1 совпадают с русскими
названиями стадий модуля (см. `/source/АП Module.md`) — выбраны
**существующие поля**, подтверждено пользователем как верный маппинг,
не гипотеза. (2) Что считать «неучитываемым» модулем в счётчике этапа:
терминальные стадии (`SUCCESS`/`FAIL`) и модули без заполненной даты
стадии — оба класса **исключены** из счёта (не попадают ни в `+`, ни
в `-`), а не искусственно засчитываются как «по графику». (3) Формат
сводки на уровне Этапа: готовая строка от бэкенда или сырые числа
+ форматирование на фронте — выбраны **сырые числа**
(`{onTrack, overdue, hasModules}` в JSON), фронт форматирует `N/M`/`БМ`
— решение приняли по ходу реализации (спека сознательно оставляла
это на усмотрение implementer'а).

**Решение.** `dashboardModuleLagDays()` в `www/api/dashboard-data.php` —
чистая функция: маппинг стадия→поле-дата (`DASHBOARD_MODULE_STAGE_DATE_FIELD`,
терминальные стадии в нём отсутствуют), `lagDays = round((дата_поля −
сегодня)/86400)`, `null` для терминальной стадии или пустой даты.
Знак: `>= 0` (включая ровно 0 — «день в день» тоже считается «по
графику») → `+`, `< 0` → `-`. `fetchDashboardData()` кладёт `lagDays`
в каждый `module`-row и агрегат `moduleLag: {onTrack, overdue,
hasModules}` в каждый `milestone`-row. Фронт (`www/js/app.js`) рендерит
колонку «Лаг» у Модуля и «М +/-» у Этапа (`N/M`, `БМ` при `!hasModules`,
`0/0` — валидный случай, когда модули есть, но ни один не участвует в
счёте).

**Почему.** Переиспользование уже существующих полей-дат — минимальный
риск и ноль изменений в БП/CRM-схеме портала; названия полей были
специально подобраны в БП под русские названия стадий, так что маппинг
не пришлось изобретать. Исключение (не «зачёт в плюс») для терминальных
и неспланированных модулей — чтобы счётчик этапа не создавал ложное
чувство благополучия там, где данных для оценки попросту нет.

**Последствия.** Финальное ревью нашло и потребовало исправить два
момента до мерджа: (1) `dashboardModuleLagDays()` изначально не ловила
исключение при невалидной/legacy-нулевой дате из CRM — один битый
модуль на любой сделке уронил бы весь `dashboard.php` (500) вместо
одной строки; добавлен `try/catch` с возвратом `null` — теперь
согласуется с остальным контрактом функции. (2) `lagDays === null`
у модуля изначально красился как «по графику» (зелёный) на фронте —
поправлено на отсутствие цветового класса. Если в будущем понадобится
похожий срыв-счётчик на уровне Сделки (не только Этапа) — это
осознанно вне охвата текущего решения (пользователь просил только
уровень Этапа), потребует отдельного решения.

## D-015 · Доводка задачи из пульта — предзаполнение названия кодом объекта, дефолтные параметры задачи зашиты константой

**Дата:** 2026-07-16 · **Статус:** принято

**Контекст.** По итогам практического использования фичи «Создать
задачу» (D-013) пользователь попросил три точечных доработки: (1)
предзаполнить поле «Название задачи» кодом объекта сделки с тире
(например `017-`), чтобы не искать/копировать код вручную; (2)
запретить исполнителю менять крайний срок задачи; (3) включить
«Проконтролировать задачу после завершения». Ни один из трёх пунктов
не был предусмотрен исходной спекой D-013.

**Варианты.** (1) Источник кода для префилла названия: `objectShortName`
(«Краткое наименование», уже используется для `UF_AUTO_824402759720`)
или `deal.code` («Код объекта», `UF_CRM_13_O_CODE`, уже отображается
в таблице и участвует в поиске `searchCode`) — выбран **`deal.code`**,
т.к. пример пользователя (`017-`) — это формат кода объекта, не
краткого наименования. (2) Параметры задачи — редактируемые в форме
модалки поля или зашитые константой на бэкенде — выбраны
**константы на бэкенде** (`www/api/task-create.php`), по аналогии с
уже зашитым тегом `TAGS` (D-013): пользователь просил фиксированное
поведение «всегда так», а не выбор при каждом создании — усложнять
форму лишним UI не нужно.

**Решение.** `openTaskModal()` (`www/js/app.js`) предзаполняет
`els.taskTitle.value = deal.code + '-'` (пусто, если код не задан).
В `task-create.php` в `fields` для `tasks.task.add` добавлены
`ALLOW_CHANGE_DEADLINE = 'N'` и `TASK_CONTROL = 'Y'` — без входа в
JSON запроса и без полей в форме. Точные имена полей и допустимые
значения (`enum`, `Y`/`N`) сверены по `tasks.task.get.fields` в
клоне `b24-rest-docs`, не угаданы (тот же принцип сверки, что и в
D-013 для `UF_AUTO_824402759720`). Проверено локальным fixture-стендом
(`php -S` + стаб `b24()`/`requireSession()`, без правки прод-кода) —
подтверждено, что оба поля уходят в тело REST-вызова, — и живой
проверкой пользователем в Б24 после деплоя.

**Почему.** Префилл кода объекта убирает ручной шаг «найти код и
вписать» — единственная причина, по которой кнопка 🎯 вообще
существует (спека D-013, §«Контекст»: не переключаться в карточку
и не заполнять вручную). Дефолтные параметры задачи зашиты
константой, а не вынесены в форму — потому что это правило
организации работы руководителя («так должно быть всегда»), а не
выбор для конкретной задачи; правки такого рода per D-013 уже
устанавливали прецедент (тег `TAGS` тоже не редактируется в форме).

**Последствия.** Если в будущем понадобится сделать эти два параметра
задачи редактируемыми (например, отдельные задачи с длинным сроком,
где смена дедлайна исполнителем уместна) — это отдельное решение,
потребует добавления полей в модалку и в тело запроса; текущее
решение такой кейс не покрывает и не проектировалось под него.
