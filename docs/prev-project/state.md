## Где мы сейчас

Онбординг завершён, дизайн-спека главной страницы утверждена:
[docs/superpowers/specs/2026-07-13-pult-rukovoditelya-design.md](superpowers/specs/2026-07-13-pult-rukovoditelya-design.md)
(вариант B — таблица сделок + KPI-плашки сверху + аккордеон
Сделка→Этап→Модуль, см. D-004 в decisions.md). Это «Пульт руководителя» —
отдельная страница (LEFT_MENU) для директора и замов портала
`alfa-prj.bitrix24.ru`: сводка по всем объектам компании (иерархия
смарт-процессов Сделка (1050) → Этап/Milestone (1054) → Модуль (1062),
плюс Оплаты (1058)) — сроки и ключевые показатели без погружения в детали
конкретной сделки. Бизнес-контекст полей, стадий и связей между смартами —
в `/source` (5 файлов: Deal, Milestone, Module, Pay, Задачи).

Инфраструктура полностью готова: деплой на VPS `rub24.blackboxbegin.space`
(см. D-005), local-app в Б24 перерегистрирован на этот домен, install-flow
пройден — токены сохранены в `data/b24-tokens.php` на сервере (`domain:
alfa-prj.bitrix24.ru`).

Бэкенд пульта реализован и проверен реальными REST-вызовами:
`www/api/dashboard-data.php` (`fetchDashboardData()`) собирает дерево
сделка→этапы→модули с агрегатами/индикаторами через batch-загрузку
`crm.item.list` по 4 сущностям (Deal/Milestone/Module/Pay, см. D-006) плюс
резолв имён разработчиков модулей через `user.get`; `www/api/dashboard.php`
— тонкий session-gated эндпоинт с `?filter=active|all|closed`. По ходу
реализации найдена и задокументирована грабля Б24
(`rules/rule-crm-item-camelcase-select.md`): `select`/ответ `crm.item.list`
— camelCase-имена полей, не классические коды из `/source`.

UI написан по контракту дизайн-спеки: KPI-плашки, тулбар (поиск по
коду/названию, чекбоксы стадий, пресеты Активные/Все/Завершённые),
сортируемая таблица сделок, аккордеон Сделка→Этап→Модуль (чистый рендер
уже загруженного дерева, без новых REST при раскрытии), баннер ошибки с
«Повторить» — `www/template.html`, `www/js/app.js`, `www/css/style.css`.

**Проверено пользователем глазами в Б24** (левое меню →
«Пульт руководителя», `alfa-prj.bitrix24.ru`) — KPI, таблица, аккордеон
отрисовались корректно на реальных данных. По итогам проверки: (1)
исправлен CSS-баг с пустым видимым `error-banner` поверх данных
(`display:flex` перебивал `hidden`), (2) по просьбе пользователя бейджи
стадий Deal/Milestone/Module теперь красятся точными цветами из воронок
Б24 (сняты `crm.status.list`, не подобраны на глаз) — см. D-007, (3)
добавлены кнопки-ссылки ↗ на карточку сущности в Б24 (Сделка/Этап/Модуль,
см. D-009), (4) KPI-плашки «Со срывом графика»/«Ожидают оплаты» теперь
показывают `N (M)` — сделки и суммарно незавершённые этапы по всем
сделкам независимо от стадии самой сделки (см. D-010). Всё задеплоено
на прод.

Известный сознательный пробел: спека описывает частичный отказ («сделки
загрузились, этапы/модули — нет» → пометка в проблемной строке, страница
не падает целиком) — `fetchDashboardData()` сейчас падает целиком при
любой REST-ошибке на любом из 4 шагов, фронт показывает полноэкранный
баннер с «Повторить». Точечная деградация не реализована — не хватило
времени в рамках этой итерации, не забыт.

**Заказчик/Ответственный + цветные счётчики стадий** (2026-07-14, ветка
`worktree-zakazchik-otvetstvenny-filtr`, спека
[docs/superpowers/specs/2026-07-14-zakazchik-otvetstvenny-filtr-design.md](superpowers/specs/2026-07-14-zakazchik-otvetstvenny-filtr-design.md),

реализовано через subagent-driven-development, 8 задач + 1 фикс, все
прошли implementer+reviewer цикл. В строке сделки — колонки «Заказчик»
(`UF_CRM_13_CUSTOMER_COMP`) и «Ответственный» (`assignedById`),
человекочитаемые (резолв через `crm.company.list`/`user.get`), плюс
клиентские dropdown-фильтры по обоим полям (без новых REST, фильтрация
по уже загруженному дереву — скрывает сделку целиком вместе с
поддеревом, т.к. у Milestone/Module своих полей заказчика/ответственного
нет). Счётчики «Этапы»/«Модули» — вместо текстовых лейблов теперь
цветные плашки-числа, упорядоченные по стадии жизненного цикла, с
`title`-подсказкой при наведении (`dashboardOrderedStageCounts()` в
`www/api/dashboard-data.php`, `countBadges()` в `www/js/app.js`). По
дороге найден и исправлен баг: поле «Заказчик Компания» на портале
хранит значение без ожидаемого документацией префикса (`"5"`, а не
`"CO_5"`) — см. D-011. Задеплоено в прод, всё проверено пользователем
глазами в Б24 (колонки, фильтры, плашки — работают).

Ветка `worktree-zakazchik-otvetstvenny-filtr` смерджена в `master`
(fast-forward) и запушена на GitHub (`raccoonspain/ap-pdir`).

**Кнопка 🎯 «Создать задачу» на строках Сделка/Этап/Модуль** (2026-07-14,
ветка `worktree-sozdanie-zadachi`, спека
[docs/superpowers/specs/2026-07-14-sozdanie-zadachi-design.md](superpowers/specs/2026-07-14-sozdanie-zadachi-design.md),
план [docs/superpowers/plans/2026-07-14-sozdanie-zadachi.md](superpowers/plans/2026-07-14-sozdanie-zadachi.md),
см. D-013): реализовано через subagent-driven-development, 4 задачи, все
прошли implementer+reviewer цикл. Кнопка на каждом уровне аккордеона
открывает свою модалку создания задачи Б24 (`tasks.task.add`) —
предзаполненное шаблонное описание, выбор исполнителя из активных
сотрудников портала, срок. Задача всегда привязывается к CRM-элементу
Сделки (`UF_CRM_TASK`, формат `T{hex(entityTypeId)}_{id}` — Сделка это
смарт-процесс `1050`, не классический Deal), автозаполняется служебное
поле «Объект наименование» (`UF_AUTO_824402759720`, источник —
подтверждённый живым REST полем `UF_CRM_13_O_SNAME`), ставится тег
«Задача из пульта», постановщиком пишется реальный пользователь (сессия
теперь хранит `userId`, резолвится через `resolveUserIdFromAuth()`).
Новый эндпоинт `www/api/task-create.php`. Задеплоено на прод, живая
проверка прошла успешно — с одной операционной граблей: после
сохранения нового scope `task` в карточке local-app уже выданный
`access_token` не подхватил его сам, потребовался форс-рефреш через
`grant_type=refresh_token` (одноразовый ручной шаг, не часть кода
приложения).

**Срыв срока на уровне Модуля — колонки «Лаг»/«М +/-»** (2026-07-15,
ветка `worktree-srok-modulya`, спека
[docs/superpowers/specs/2026-07-15-srok-modulya-design.md](superpowers/specs/2026-07-15-srok-modulya-design.md),
план [docs/superpowers/plans/2026-07-15-srok-modulya.md](superpowers/plans/2026-07-15-srok-modulya.md),
см. D-014): на уровне Этапа срыв уже считался готовым полем БП
(`UF_CRM_15_MST_CONTR_PLAN`), на уровне Модуля такой автоматизации нет —
реализовано в приложении. Новая чистая функция `dashboardModuleLagDays()`
в `www/api/dashboard-data.php` берёт дату-план текущей стадии модуля из
уже существующих полей (`UF_CRM_19_MOD_RUN/CHECK/CREATE/EDIT/WAIT/APPROVE`,
названия полей 1:1 совпадают с русскими названиями стадий) и сравнивает
с сегодня; `null` для терминальных стадий (Согласовано/Аннулировано) и
для ещё не спланированной даты — такие модули не участвуют в счёте.
`fetchDashboardData()` отдаёт `lagDays` у модуля и агрегат
`moduleLag: {onTrack, overdue, hasModules}` у этапа. UI: колонка «Лаг»
у Модуля, колонка «М +/-» у Этапа (`N/M`, `БМ` без модулей, `0/0` —
валидный случай). Реализовано через subagent-driven-development, 3
задачи, все прошли implementer+reviewer цикл; финальное whole-branch
ревью нашло и потребовало исправить: (1) незащищённый разбор даты мог
уронить весь `dashboard.php` при одной битой/legacy-нулевой дате в CRM
— добавлен `try/catch`, (2) `lagDays === null` красился как «по
графику» (зелёным) — убран цветовой класс. По ходу живой проверки
пользователь также нашёл и попросил исправить конфликт специфичности
CSS: заголовки `.num` в подтаблицах выравнивались влево, а значения —
вправо (задевало не только новые колонки, но и старые «Цена»/«Дни
КП-План»). Всё задеплоено на прод, подтверждено пользователем.

Ветка `worktree-srok-modulya` смерджена в `master` (fast-forward после
каждой задачи в ходе реализации, а не одним финальным мерджем).

**Доводка модалки «Создать задачу»** (2026-07-16, прямо в `master`):
три точечных правки по фиче из D-013. (1) Поле «Название задачи» в
`openTaskModal()` (`www/js/app.js`) теперь предзаполняется кодом
объекта сделки (`deal.code`) с тире, например `017-` — руководителю
остаётся дописать суть задачи. (2) В `www/api/task-create.php` в
`tasks.task.add` добавлены два дефолтных параметра задачи:
`ALLOW_CHANGE_DEADLINE = 'N'` («Разрешить исполнителю менять крайний
срок» — нет) и `TASK_CONTROL = 'Y'` («Проконтролировать задачу после
завершения» — да); имена полей и допустимые значения (`Y`/`N`, enum)
сверены по `tasks.task.get.fields` в клоне `b24-rest-docs`, не угаданы.
Backend-логика проверена локальным fixture-стендом (`php -S` +
стаб-заглушки `b24()`/`requireSession()`, по образцу паттерна из спеки
D-013) — подтверждено, что оба поля реально уходят в тело
`tasks.task.add`. Задеплоено (`bash deploy.sh`), пользователь
подтвердил живьём в Б24 — префилл названия и оба параметра задачи
работают как задумано. Решение зафиксировано в D-015.
