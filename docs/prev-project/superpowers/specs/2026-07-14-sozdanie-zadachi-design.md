# Кнопка 🎯 «Создать задачу» (Сделка/Этап/Модуль)

**Дата:** 2026-07-14 · **Статус:** утверждено пользователем

## Контекст

Руководителю нужно быстро ставить задачу «руками» по любой проблемной
сущности пульта (Сделка/Этап/Модуль), не переключаясь в саму карточку
Б24 и не заполняя привязки/описание вручную. Отдельно от этого в модуле
уже есть автоматическая постановка задач через бизнес-процессы Б24
(см. `source/АП Задачи.md`, `source/АП Module.md`) — эта фича с ними не
пересекается технически, но по продукту должна визуально отделяться
(см. тег в §4).

## 1. UI: кнопка и модалка

Кнопка 🎯 — в ячейке названия, рядом с уже существующей ↗-ссылкой на
карточку в Б24, на всех трёх уровнях аккордеона: `dealRowHtml()`,
`milestoneRowHtml()`, `moduleRowHtml()` (`www/js/app.js`). `moduleRowHtml()`
сейчас принимает только `mod` — нужно передавать также `deal`/`milestone`
(по образцу `milestoneRowHtml(deal, milestone)`), чтобы у кнопки модуля
были под рукой все названия для текста описания и `id` сделки для
привязки.

Клик открывает модалку (свой HTML/CSS/JS, без сторонних библиотек —
в стиле остального фронта):

- **Название** — пустой `input`, автофокус, обязательное.
- **Описание** — `textarea`, предзаполнена шаблонным текстом (см. §2),
  редактируемая.
- **Исполнитель** — `<select>` со всеми активными сотрудниками портала,
  обязательное.
- **Срок** — `input type="date"`, необязательное.
- Кнопки «Создать» / «Отмена».

После успешного создания: модалка закрывается, тост «Задача создана» со
ссылкой «Открыть в Битрикс24» (`BX24.openPath('/company/personal/user/0/tasks/task/view/{ID}/')`).
Ошибка REST — текст ошибки прямо в модалке (тот же паттерн, что и текст
ошибки дашборда), форма остаётся заполненной для повтора отправки.

## 2. Текст описания (генерится на фронте, редактируется руководителем)

Из уже загруженного дерева, без новых REST-вызовов на открытие модалки:

- Сделка: `Задача к сделке «{deal.title}»\n`
- Этап: `Задача к этапу «{milestone.title}» сделки «{deal.title}»\n`
- Модуль: `Задача к модулю «{module.title}» этапа «{milestone.title}» сделки «{deal.title}»\n`

## 3. Привязки и автозаполнение (не видны в форме, заполняются в фоне)

- **Привязка к CRM** (`UF_CRM_TASK`) — всегда к Сделке, независимо от
  того, на каком уровне нажата кнопка. Формат значения для привязки
  задачи к сделке — сверить перед реализацией по
  `/home/deploy/refs/b24-rest-docs` (не угадывать, см. `rules/rule-crm-item-universal-api.md`
  за прецедентом такой же сверки).
- **`UF_AUTO_824402759720` («Объект наименование»)** — значение поля
  «Краткое наименование» Сделки. В `source/АП Deal.md` код поля
  `UF_CRM_13_SNAME`, в `source/АП Задачи.md` пример автозамены
  ссылается на `UF_CRM_13_O_SNAME` — расхождение в исходниках. Перед
  реализацией сверить точный код через
  `crm.item.fields.json?entityTypeId=1050`, не гадать.
- **Постановщик (`CREATED_BY`)** — текущий пользователь, открывший
  Пульт.

## 4. Тег задачи

К создаваемой задаче добавляется тег `Задача из пульта` (поле `TAGS`
в `tasks.task.add`, массив строк) — чтобы можно было в Б24 отделить
задачи, поставленные вручную через Пульт, от автоматических (роботы
БП) и от задач, заведённых вручную внутри самой карточки модуля.
Значение тега зашито константой на бэкенде, не редактируется в форме.

## 5. Бэкенд: постановщик через сессию

Сейчас сессия (`www/api/session.php`, `createSession()`) хранит только
`token`/`createdAt`/`expiresAt` — id пользователя нигде не сохраняется,
хотя `authId` уже резолвится в `tryCreateSessionFromB24Post()` для
admin-gate через `b24IsPortalAdmin()`. Меняем:

- В `tryCreateSessionFromB24Post()` после прохождения admin-gate —
  дополнительный вызов `resolveUserIdFromAuth($authId, $stored)`
  (уже существует в `www/api/b24.php`), результат передаётся в
  `createSession()`.
- `createSession(?string $userId)` сохраняет `userId` в структуру
  сессии наравне с `token`.
- `requireSession()` возвращает сессию как раньше (без изменения
  сигнатуры) — вызывающий код (`task-create.php`) читает
  `$session['userId']`.
- Если `userId` не резолвился (REST-сбой на установке сессии) —
  сохраняем `null`, `task-create.php` в этом случае отдаёт понятную
  ошибку («не удалось определить текущего пользователя»), не пишет
  задачу без постановщика.

## 6. Бэкенд: новый эндпоинт `www/api/task-create.php`

Session-gated (`requireSession()`) как `dashboard.php`. Принимает JSON:
`entityDealId`, `title`, `description`, `responsibleId`, `deadline`
(опционально). Вызывает `tasks.task.add` одним REST-вызовом:

```
TITLE           = title
DESCRIPTION     = description
RESPONSIBLE_ID  = responsibleId
CREATED_BY      = session.userId
DEADLINE        = deadline (если передан)
UF_CRM_TASK     = [привязка к entityDealId] — формат см. §3
UF_AUTO_824402759720 = objectShortName сделки (см. ниже, откуда берём)
TAGS            = ['Задача из пульта']
```

`objectShortName` не запрашивается заново отдельным REST-вызовом —
поле «Краткое наименование» (код подтверждён живой проверкой
`crm.item.fields` на портале: `UF_CRM_13_O_SNAME`, camelCase
`ufCrm13OSname`) добавляется в `$dealSelect` в `dashboard-data.php`,
попадает в дерево, фронт передаёт его значение в теле запроса к
`task-create.php` вместе с `entityDealId` (backend не обязан заново
читать сделку из CRM).

Ответ: `{ taskId }` либо `{ error }` с 4xx/5xx. `taskUrl` не нужен —
фронт сам строит путь до задачи через `BX24.openPath()` по `taskId`
(см. §1, кнопка «Открыть в Битрикс24»).

## 7. Бэкенд: список исполнителей

Список активных сотрудников для `<select>` — один REST-вызов
`user.get` (`ACTIVE => true`), встроенный в уже существующий
`fetchDashboardData()` в `dashboard-data.php`, результат отдаётся в
JSON вместе с деревом одним полем `users: [{id, name}]`. Без отдельного
round-trip при открытии модалки (тот же принцип, что и у остального
дерева — «раскрытие/действия без новых REST», см. `docs/superpowers/specs/2026-07-13-pult-rukovoditelya-design.md`).

## 8. Скоуп Б24

`tasks.task.add` требует scope `task`, которого сейчас нет
(`crm user placement`, см. `docs/project-brief.md`). По конвенции
проекта (лениво по факту запроса, `CLAUDE.md`) — `task` добавляется
в карточку local-app в Б24 вручную пользователем в админке портала;
после сохранения карточки может понадобиться одно переоткрытие
приложения, чтобы токен обновился с новым скоупом. `docs/project-brief.md`
(поле «Скоупы Б24») обновляется на `crm user placement task` по факту
подтверждения, что задача создаётся успешно.

## Тестирование

Ручная проверка по образцу уже применявшегося в проекте паттерна
(D-006, D-010): фикстура-заглушка `B24` (не коммитится) на
`task-create.php` и на изменение `createSession()`/`requireSession()`
flow — проверить, что `UF_CRM_TASK`/`UF_AUTO_824402759720`/`TAGS`/
`CREATED_BY` уходят в `tasks.task.add` с ожидаемыми значениями при всех
трёх уровнях клика (Сделка/Этап/Модуль). Реальным REST — на следующем
деплое на `alfa-prj.bitrix24.ru`: создать задачу по кнопке с каждого из
трёх уровней, проверить в Б24 карточке задачи — привязку, тег,
постановщика, значение `UF_AUTO_824402759720`.

Фронтенд — вручную глазами в браузере (dev-сервер или прод после
деплоя): кнопка на всех трёх уровнях, предзаполненный текст описания,
список исполнителей, создание, тост со ссылкой, поведение при ошибке
REST — по существующей практике проекта (нет JS-тестов, UI проверяется
глазами, см. `CLAUDE.md`).

## Не входит в объём (не поднято пользователем)

- Прикрепление файлов/чек-листов к задаче при создании — не запрошено,
  дособирается руководителем уже в самой задаче в Б24.
- Кэширование списка исполнителей между запросами — как и остальной
  пульт, кэш не заведён (см. D-004).
- Валидация «исполнитель существует и активен» на бэкенде сверх того,
  что даёт сам `tasks.task.add` — не запрошено.
