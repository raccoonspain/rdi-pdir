# Пульт руководителя — дизайн

**Дата:** 2026-07-13 · **Статус:** утверждено, готово к планированию реализации

## Назначение

Отдельная страница в Битрикс24 (портал `alfa-prj.bitrix24.ru`, плейсмент `LEFT_MENU`,
не вкладка в карточке сделки) для директора и его замов. Одним экраном показывает
все объекты компании (иерархия смарт-процессов Сделка → Этап (Milestone) → Модуль)
со сроками и ключевыми показателями, без необходимости открывать каждую сделку.
Бизнес-контекст полей, стадий и связей между смартами — в `/source` (5 файлов:
Deal, Milestone, Module, Pay, Задачи).

Скоуп реализации: только чтение и визуализация данных. Изменение данных в Б24
(смена стадий, редактирование полей) через пульт не предусмотрено — для этого
директор идёт в саму сделку.

## Сущности (см. `/source` для полного списка полей)

| Смарт | entityTypeId | Роль |
|---|---|---|
| Сделка (Deal) | 1050 | Объект/контракт верхнего уровня |
| Этап (Milestone) | 1054 | Дочерний по отношению к Deal (`PARENT_ID_1050`) |
| Модуль (Module) | 1062 | Дочерний по отношению к Milestone (`PARENT_ID_1054`) и Deal |
| Оплата (Pay) | 1058 | Дочерний по отношению к Milestone, привязан к сумме к оплате/оплачено |

Порядок стадий Deal (бизнес-порядок, не алфавитный):
1. Подписание (`NEW`)
2. Авансирование (`UC_WRET3K`)
3. Работа (`CLIENT`)
4. Закрытие (`PREPARATION`)
5. Завершено (`SUCCESS`)
6. Разрыв (`FAIL`)

## Архитектура загрузки данных

**Один backend-эндпоинт** `www/api/dashboard.php`, вызывается фронтом при
открытии страницы (без cron — см. `docs/state.md`). На каждый запрос:

1. `crm.item.list` entityTypeId=1050 — сделки, отфильтрованные по активному
   набору стадий (см. «Фильтры» ниже).
2. `crm.item.list` entityTypeId=1054 — этапы всех найденных сделок (фильтр
   по `PARENT_ID_1050 IN (...)`), включая поля `Дни КП-План`, `Последняя
   активность`, `Дата активности`, оплачено/остатки.
3. `crm.item.list` entityTypeId=1062 — модули всех найденных сделок
   (фильтр по `PARENT_ID_1050 IN (...)`), включая стадию, разработчика,
   даты, последнюю активность.

Каждый из трёх шагов постранично (Б24 REST отдаёт до 50 записей за вызов)
собирается через `batch()` — несколько страниц `list` в одном batch-запросе,
не циклом отдельных вызовов (см. `rules/rule-b24-rest-batch-not-loop.md`).
При ожидаемом объёме (≈50 сделок × 4 этапа × 20 модулей ≈ 4000 модулей)
это укладывается в 2–3 batch-запроса.

Backend сразу считает агрегаты на сервере и отдаёт фронту одним JSON готовое
дерево `сделка → этапы → модули` вместе с вычисленными полями (счётчики по
стадиям, индикаторы срыва графика/ожидания оплаты, отставание в днях).
Раскрытие строки в аккордеоне на фронте — чистый рендер уже полученных
данных, никаких дополнительных REST-вызовов.

Кэш не используется на первой итерации. Логика сборки данных вынесена в
одну чистую функцию `fetchDashboardData()` в `api/lib.php`, чтобы при росте
объёма данных её можно было обернуть в файловый кэш с TTL (по аналогии со
`store.php`), не трогая остальной код — см. также `data/`-конвенции в
`CLAUDE.md`.

## Макет страницы (вариант B — таблица + KPI-сводка)

Утверждённый вариант из трёх черновиков (A — плоская таблица без сводки,
B — таблица + KPI-плашки сверху, C — группировка по стадии). Выбран B.

Сверху экрана — строка из 4 плашек-агрегатов по текущей выборке (с учётом
активных фильтров):
- **Активных сделок** — количество строк в текущей выборке
- **Сумма контрактов** — сумма `Цена контракта` по выборке
- **Со срывом графика** — количество сделок с индикатором 🔴
- **Ожидают оплаты** — количество сделок с индикатором 🟡

Под плашками — тулбар (поиск + фильтры), под ним — таблица сделок.

## Таблица сделок

Одна строка = одна сделка. Колонки (в порядке слева направо):

| Колонка | Источник | Примечание |
|---|---|---|
| (иконка раскрытия) | — | клик по строке разворачивает этапы |
| Код | `UF_CRM_13_O_CODE` | моноширинный шрифт |
| Название | `TITLE` | |
| Стадия | `STAGE_ID` | цветной бейдж, человекочитаемое название |
| Цена | `UF_CRM_13_O_COST` | выравнивание по правому краю |
| Остаток | `UF_CRM_13_O_BALANCE` | выравнивание по правому краю; колонка «Оплачено» в таблицу **не** выводится — сумма уже видна в KPI-сводке, точная цифра оплаты смотрится при раскрытии сделки |
| Индикаторы | вычисляется | 🔴/🟡, см. «Правила статуса» |
| Этапы | агрегат по Milestone | счётчик по стадиям, например «2 раб · 1 опл» |
| Модули | агрегат по Module | счётчик по стадиям, например «7 раб · 3 закр» |
| Отставание | `UF_CRM_15_MST_CONTR_PLAN` (Milestone) | худшее (наиболее отрицательное) значение среди открытых этапов сделки, ± дней |

Для сделок на стадии **Подписание** или **Авансирование** колонки
«Этапы» / «Модули» / «Отставание» пустые (`—`) — этапы физически ещё не
заведены на этих стадиях (см. `/source` → «Как вообще проходит работа по
исполнению контракта»).

## Сортировка

Клик по заголовку колонки переключает сортировку (по возрастанию/убыванию):
- **Код** — лексикографически
- **Стадия** — по бизнес-порядку 1→6 (см. «Сущности»), не по алфавиту
  технического кода стадии

## Фильтры

- Текстовый поиск по **Код** (подстрока)
- Текстовый поиск по **Название** (подстрока)
- Мультивыбор по **Стадия** (чекбоксы по всем 6 стадиям)
- Три кнопки-пресета поверх чекбоксов: **Активные** (дефолт — все стадии
  кроме Завершено и Разрыв) / **Все** / **Только завершённые** (Завершено +
  Разрыв)

Дефолт — «Активные»: сделано ради скорости загрузки (шаг 1 архитектуры
запрашивает у Б24 сразу отфильтрованный список, не все сделки).

## Правила статуса (индикаторы)

Два независимых сигнала, отображаются как отдельные цветные точки в
колонке «Индикаторы» строки сделки (без заливки всей строки цветом):

- 🔴 **Срыв графика** — `Дни КП-План` < 0 у любого открытого (не
  Завершено/Разрыв) этапа сделки.
- 🟡 **Ожидание оплаты** — есть этап в стадии «Оплата» ИЛИ есть запись Pay
  в стадии «Счёт отправлен». Намеренно отдельный от срыва графика цвет —
  чтобы не путать денежные события со срывом срока реализации.

Индикаторы не показываются для сделок на стадии Подписание/Авансирование
(нет этапов, см. выше) и вычисляются только на активных (не
Завершено/Разрыв) этапах.

## Аккордеон: сделка → этапы → модули

Клик по строке сделки разворачивает вложенную таблицу этапов: Номер,
Название, Стадия, Дни КП-План, Последняя активность + дата. Клик по
строке этапа (внутри развёрнутого блока) разворачивает вложенную таблицу
его модулей: Номер, Название, Стадия, Разработчик, даты по вехам,
Последняя активность.

Все данные уже загружены на шаге 1 (единый набор batch-запросов при
открытии страницы) — раскрытие не делает новых REST-вызовов, это чистый
клиентский рендер уже полученного дерева.

## Обработка ошибок

- REST-вызов упал целиком (истёк токен, недостаточно прав, таймаут) —
  страница показывает сообщение об ошибке с кнопкой «Повторить», не белый
  экран.
- Частичный успех (сделки загрузились, этапы/модули — нет) — сделки
  показываются с пометкой «данные по этапам недоступны» в проблемной
  строке; страница не падает целиком.

## Вне скоупа этой итерации

- Изменение/редактирование данных через пульт (только просмотр)
- Кэширование (сознательно отложено, см. «Архитектура загрузки данных»)
- Уведомления/алерты о просрочках (пульт — по требованию открытия
  страницы, не push)
- Мобильная адаптация (не обсуждалась, дефолт — desktop-таблица)
