# Срыв срока на уровне Модуля

**Дата:** 2026-07-15

## Проблема

Сейчас в пульте срыв сроков считается только на уровне Этапа (Milestone) —
через готовое поле `UF_CRM_15_MST_CONTR_PLAN` (лаг в днях, вычисляет БП).
На уровне Модуля (Module) такой автоматизации в бизнес-процессе нет и
соответствующего готового поля тоже нет — расчёт нужно делать в приложении.

## Откуда берём дату завершения стадии модуля

В `/source/АП Module.md` подтверждено: у каждой нетерминальной стадии
модуля есть своё поле-дата, название которого 1:1 совпадает с русским
названием стадии. Это и есть «дата завершения текущей стадии»:

| Стадия (код) | Название         | Поле-дата                  |
|---|---|---|
| `NEW`         | Запуск         | `UF_CRM_19_MOD_RUN` (вычисляется)      |
| `PREPARATION` | Рассмотрение   | `UF_CRM_19_MOD_CHECK` (вручную)        |
| `CLIENT`      | Разработка     | `UF_CRM_19_MOD_CREATE` (вручную)       |
| `UC_WI1QUU`   | Корректировка  | `UF_CRM_19_MOD_EDIT` (вычисляется)     |
| `UC_MTO1QJ`   | Ожидание       | `UF_CRM_19_MOD_WAIT` (вычисляется)     |
| `UC_DFWFJU`   | Согласование   | `UF_CRM_19_MOD_APPROVE` (вычисляется)  |
| `SUCCESS`     | Согласовано    | — (терминальная, не считаем)           |
| `FAIL`        | Аннулировано   | — (терминальная, не считаем)           |

Маппинг подтверждён пользователем как верный (не гипотеза).

## Расчёт лага модуля (`lagDays`)

Для каждого модуля:

1. Если стадия терминальная (`SUCCESS`/`FAIL`) → `lagDays = null`, модуль
   **не участвует** в счётчике «М +/-» на уровне этапа.
2. Иначе берём поле-дату текущей стадии из таблицы выше.
   - Поле пустое (стадию ещё не спланировали) → `lagDays = null`, модуль
     **не участвует** в счётчике.
   - Поле заполнено → `lagDays = round((дата_поля − сегодня) / 86400)` в
     днях.
3. Знак и порог — та же конвенция, что уже используется для Этапа
   (`ufCrm15MstContrPlan`):
   - `lagDays >= 0` (включая ровно 0 — «день в день по графику») → модуль
     **по графику**.
   - `lagDays < 0` → модуль **со срывом** внутреннего срока стадии.

## Колонка «М +/-» на уровне Этапа

Для каждого Этапа (Milestone) считаем по всем его модулям, у которых
`lagDays !== null`:

- `onTrack` = кол-во модулей с `lagDays >= 0`
- `overdue` = кол-во модулей с `lagDays < 0`
- Строка колонки: `"{onTrack}/{overdue}"`, например `5/2`.

Особые случаи:

- У этапа вообще нет заведённых модулей → `БМ` («без модулей»).
- У этапа есть модули, но ни один не участвует в счёте (все терминальные
  и/или без даты) → `0/0` (счётчик существует, просто оба числа нулевые).
