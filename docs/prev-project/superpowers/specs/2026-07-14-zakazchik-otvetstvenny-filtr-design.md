# Заказчик/Ответственный (колонки + фильтры) + редизайн счётчиков Этапы/Модули

**Дата:** 2026-07-14 · **Статус:** утверждено пользователем

## Контекст

Следующий шаг по фидбеку пользователя от 2026-07-13 (см. `docs/state.md`):
показать в строке сделки Заказчика и Ответственного, с фильтром по обоим
полям. По ходу обсуждения размещения (новые колонки — таблица и так на
10 колонок) пользователь попросил заодно почистить существующие колонки
«Этапы»/«Модули»: текстовые лейблы стадий («2 заверш · 1 ожид») съедают
место, нужен компактный цветной вид с сохранением читаемости стадии.

## Поля-источники (подтверждено пользователем)

- Заказчик: `UF_CRM_13_CUSTOMER_COMP` на Deal (1050) → camelCase
  `ufCrm13CustomerComp`. Тип «Привязка к элементам CRM», одиночная
  (не `isMultiple`, в отличие от `UF_CRM_13_CUSTOMER_PERSON`). Формат
  значения — `{PREFIX}_{ID}`, для компании `PREFIX = CO` (не `C` — тот
  занят под Контакт), пример `CO_123`. См.
  `/home/deploy/refs/b24-rest-docs/api-reference/crm/data-types.md`
  (раздел «Формат значений для пользовательского поля „Привязка к
  элементам CRM“»).
- Ответственный: `ASSIGNED_BY_ID` → camelCase `assignedById`, стандартное
  поле CRM, числовой ID пользователя.

## 1. Бэкенд: резолв и новые поля сделки

`www/api/dashboard-data.php`:

- `dealSelect` дополняется `ufCrm13CustomerComp`, `assignedById`.
- Новая функция `dashboardParseCrmBindingId(?string $value): ?int` —
  разбирает `"CO_123"` → `123` (по `_`, берёт хвост, приводит к int;
  `null`/пусто/не совпал формат → `null`). Не привязана к конкретному
  префиксу — префикс не проверяется строго (на случай, если поле
  когда-то станет множественным или сменит тип, парсинг не должен упасть
  с ошибкой, просто не даст ID).
- Новая функция `dashboardResolveCompanyNames(B24 $b24, array $companyIds): array`
  — один REST-вызов `crm.company.list(filter: {ID: [...]}, select: [ID, TITLE])`,
  возвращает `[id => title]`. По образцу `dashboardResolveUserNames()`
  (тот же паттерн: `array_unique`/`array_filter`, пустой список → `[]`
  без похода в REST, эндпоинт не в `crm.item.*`, так как компания —
  не смарт-процесс, а стандартная CRM-сущность).
- Ответственного резолвим существующей `dashboardResolveUserNames()` —
  добавляем `assignedById` сделок в тот же список ID, что уже собирается
  для разработчиков модулей (`ufCrm19ModCreatorUser`), один общий
  REST-вызов `user.get` на все ID сразу (не два отдельных).
- В строку сделки (`$dealRows[]`) добавляются:
  - `customerName` — `$companyNames[$companyId] ?? null`
  - `assigneeName` — `$developerNames[(string)$deal['assignedById']] ?? null`
    (переиспользуем ту же resolve-мапу, имя останется общим для
    ответственных по сделкам и разработчиков модулей — оба User-сущности)

Ошибка резолва (компания/юзер удалены, `crm.company.list`/`user.get`
не находит ID) — тот же паттерн, что уже есть в `dashboardResolveUserNames()`:
ID просто не попадает в мапу, `?? null` даёт «—» на фронте, страница не
падает.

## 2. Бэкенд: счётчики этапов/модулей — новый формат

Сейчас `dealMilestoneCounts`/`dealModuleCounts` — `{shortLabel: count}`,
порядок — порядок появления в данных (не гарантирован). Меняем на
упорядоченный список объектов, порядок = порядок массива
`DASHBOARD_MILESTONE_STAGES`/`DASHBOARD_MODULE_STAGES` (он уже совпадает
с жизненным циклом стадии — `NEW → ... → SUCCESS/FAIL`, тот же порядок,
что использует `DASHBOARD_DEAL_STAGES['order']` для сделок):

```php
[
  ['stageCode' => 'CLIENT', 'name' => 'В работе', 'color' => '#10e5fc', 'count' => 2],
  ['stageCode' => 'SUCCESS', 'name' => 'Завершено', 'color' => '#00ff00', 'count' => 1],
]
```

Стадии с `count === 0` в список не попадают (как и сейчас — лейбл не
появлялся, если счётчика не было). Реализация: агрегация в `$dealMilestoneCounts`/
`$dealModuleCounts` остаётся по `$mStageCode`/`$modStageCode` (не по
короткому лейблу — короткие лейблы `DASHBOARD_MILESTONE_SHORT_LABELS`/
`DASHBOARD_MODULE_SHORT_LABELS` для этой колонки больше не нужны, но в
константах не трогаем — вдруг ещё где-то используются); на выходе из
цикла по сделке — маппинг в упорядоченный список одним проходом по
`DASHBOARD_MILESTONE_STAGES`/`DASHBOARD_MODULE_STAGES` (порядок
гарантирован PHP-семантикой ассоц. массива — вставлен в порядке
объявления констант).

## 4. Фронтенд: фильтры Заказчик/Ответственный

Тулбар (`www/template.html`, `.toolbar`): два новых `<select>` рядом с
существующими `#search-code`/`#search-title` — `#filter-customer`,
`#filter-assignee`, первая опция `<option value="">Заказчик (все)</option>`
/ `«Ответственный (все)»`.

`www/js/app.js`:
- Опции строятся при каждом `loadData()` (после получения `state.deals`,
  до первого `renderTable()`) — `Array.from(new Set(state.deals.map(d
  => d.customerName).filter(Boolean))).sort()`, аналогично для
  `assigneeName`. Пересчитываются заново при смене пресета (тот же
  жизненный цикл, что у `renderStageChecks()`).
- `state.filterCustomer`/`state.filterAssignee` — строка, `''` = без
  фильтра. Обновляются по `change` на дропдаунах, `renderTable()`.
- `getFilteredSortedDeals()`: доп. условия `if (state.filterCustomer &&
  d.customerName !== state.filterCustomer) return false;` (аналогично
  assignee) — по И с существующими условиями (код/название/стадия).
  Точное совпадение (не substring, как у текстовых полей поиска) — это
  выбор из значений, а не свободный ввод.

Поддерева (Milestone/Module) своих значений заказчика/ответственного не
имеют (проверено — в `/source` `АП Milestone.md`/`АП Module.md` таких
полей нет) — фильтрация уровня сделки автоматически скрывает/показывает
всё поддерево целиком, дополнительной логики пересборки дерева не нужно.

## 5. Фронтенд: цветные плашки счётчиков

Новая функция `countBadges(counts)` (по образцу существующей
`stageBadge()`/`indicatorDots()`) — рендерит список плашек:

```html
<span class="count-badge" style="background:#10e5fc;color:#161616"
      title="В работе">2</span>
```

`title` — атрибут, даёт нативный tooltip браузера с читаемым названием
стадии при наведении (закрывает запрос пользователя на «иконографику» —
отдельная статичная легенда не заводится, каждая плашка самодостаточна).
Порядок плашек — порядок элементов в пришедшем с бэкенда списке (уже
упорядочен на сервере, см. п.2). Цвет текста — переиспользуем существующую
`contrastTextColor()`.

`fmtCounts()` (текущая функция форматирования `{label: count}` в строку)
заменяется на `countBadges()`, работающую с новым форматом списка.
Старая `fmtCounts` удаляется (больше не используется).

`www/css/style.css`: новый класс `.count-badge` — компактный `inline-flex`,
`padding` меньше, чем у `.stage-badge` (это просто число, не слово),
`border-radius` небольшой, моноширинный шрифт (`--font-mono`, см. D-008 —
числа/лейблы уже моно в проекте).
