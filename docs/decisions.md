# Журнал решений

> Здесь — **почему** проект устроен так, а не иначе. Каждое значимое
> решение: что выбрали, какие были варианты, почему именно это.
> Файл **только дописывается** (новые записи — вниз). Старые записи не
> удаляем: если решение отменили — добавляем новую запись «отменяет D-NNN».
>
> Зачем: чтобы следующий исполнитель не «переоткрывал» уже закрытые вопросы
> и не ломал то, что сделано осознанно. Один абзац здесь экономит часы.

Записывай решение, когда выбор был **неочевидным** или у него есть
последствия: скоупы, структура данных, плейсменты, обход бага и т.п.

> Решения первого проекта на этом движке (клиент «Альфа», `ap-pdir`,
> D-001…D-015 — включая инженерные паттерны, которые скорее всего повторятся
> и здесь: camelCase у `crm.item.list`, hex-префикс CRM-привязки для
> смарт-процессов, батчи вместо foreach, цвета стадий через `crm.status.list`)
> — в [prev-project/decisions.md](./prev-project/decisions.md). Нумерация
> `D-NNN` в этом файле начинается заново с D-001 — не продолжение той
> нумерации.

## D-001 · Деплой — после переноса бизнес-кода и привязки GitHub, не сразу после онбординга
**Дата:** 2026-07-22 · **Статус:** принято

**Контекст.** После прохождения онбординга (название, портал, плейсмент, cron)
ассистент предложил сразу `bash deploy.sh` + регистрацию local-app — по
аналогии со сценарием «стартуем» для проекта с нуля из `CLAUDE.md`.
**Варианты.** (1) Деплоить сразу — OAuth/install-flow можно проверить и без
готового бизнес-кода; (2) сначала перенести код под сущности нового портала
и получить ссылку на GitHub-репозиторий, деплоить после.
**Решение.** Вариант 2. Онбординг ≠ готовность к деплою для проекта-
тиражирования: `www/api/dashboard-data.php`/`task-create.php` всё ещё
содержат entityTypeId/UF-коды первого клиента («Альфа»), это надо переписать
под сущности `ooordi.bitrix24.ru` (см. `/source/RDI *.md`). Плюс нет ссылки
на GitHub-репозиторий для этого проекта — её нужно получить и привязать
`origin` до деплоя, чтобы история проекта не потерялась.
**Почему.** Деплой скелета с чужими ID полей — рабочий, но бессмысленный шаг:
он не даёт протестировать ничего специфичного для нового клиента, а без
git-репозитория на GitHub любой снимок остаётся только локальным.
**Последствия.** Для будущих тиражирований этого шаблона: онбординг (анкета)
и готовность к деплою — разные вехи. Деплою предшествуют: (а) перенос
бизнес-кода под сущности нового портала, (б) привязка GitHub origin.

## D-002 · Бизнес-код перенесён по документам разработчика, без живой REST-сверки; цвета стадий — плейсхолдер
**Дата:** 2026-07-22 · **Статус:** принято

**Контекст.** Нужно было переписать `www/api/dashboard-data.php` и
`www/api/task-create.php` под сущности клиента «АУП» (entityTypeId и UF-коды
полей четырёх смарт-процессов на портале `ooordi.bitrix24.ru`). Local-app на
этом портале ещё не зарегистрирован — живой REST-доступ (`crm.item.fields`,
`crm.status.list`) недоступен (в отличие от первого клиента, где, например,
D-007 фиксирует цвета стадий, снятые именно так).
**Варианты.** (1) Ждать install-flow и снимать entityTypeId/UF-коды/цвета
живыми REST-вызовами, как это было сделано для первого клиента; (2) взять
за источник истины документы `/source/RDI *.md` — написаны самим
разработчиком портала при создании смарт-процессов, содержат все UF-коды и
описание бизнес-логики (веса планирования, пересчёт остатков, привязки).
**Решение.** Вариант 2 для entityTypeId и UF-кодов полей — они переписаны из
`/source` напрямую. Для цветов стадий (`color` в `DASHBOARD_*_STAGES`,
которых в `/source` нет) — плейсхолдер: значения перенесены по смыслу с
воронки первого клиента (чёрный → жёлтый → голубой → зелёный →
ярко-зелёный/красный по порядку стадий), не более того.
**Почему.** Пользователь подтвердил, что коды в `/source` — те же, что он
сам вписывал при создании полей в Б24, то есть источник авторитетный
(в отличие от документации первого клиента, где часть UF-кодов пришлось
переснимать живым REST из-за расхождений). Ждать install-flow ради каждой
правки кода было бы циклом деплой-правка-передеплой; переносить весь код
из /source сразу и сверить один раз после install — быстрее и не менее
надёжно. Цвета же в `/source` физически отсутствуют — их взять неоткуда,
кроме как с портала.
**Последствия.** После install-flow нужен один REST-проход: (1) сверить
`crm.item.fields` по всем четырём entityTypeId — на случай, что реальные
коды разошлись с `/source` (человеческий фактор при ручном документировании);
(2) `crm.status.list` — заменить плейсхолдерные `color` на реальные из
воронки. Оба пункта — в docs/state.md «Следующие шаги». До этого прохода
цвета плашек в UI могут не совпадать с настройками стадий в Б24 (это
косметика, не блокер функциональности).
