## Где мы сейчас

Приложение «Пульт АУП» установлено на портале `ooordi.bitrix24.ru` и рабочее
— пользователь подтвердил скриншотом: Пульт руководителя открылся из левого
меню, показывает 2 активные сделки (`<9990> ЖК Солнечный`, `<9991> ЖК Мечта`)
с реальными кодами, стадией «Работа», суммами контрактов/остатков, счётчиками
этапов и модулей по стадиям, индикатором срыва графика. Это живое
подтверждение, что entityTypeId и все UF-коды, перенесённые из `/source/RDI
*.md` без предварительной REST-сверки (см. D-002), совпали с реальными
полями на портале — иначе поля были бы пустыми/null, а не заполненными
реальными значениями.

Код (`www/`), инженерные решения и грабли Б24 (`rules/`) перенесены из
`ap-pdir` как есть — архитектура и REST-паттерны те же (тот же движок
«Пульта»: Сделка → Этап → Модуль). Бизнес-контекст сущностей портала «АУП» —
в `/source/RDI *.md`: Сделка (entityTypeId=1038), Этап/Milestone (1042),
Модуль (1050), Оплаты (1046).

История первого проекта (для клиента «Альфа», портал `alfa-prj.bitrix24.ru`)
— в [docs/prev-project/](./prev-project/) (`state.md`, `changelog.md`,
`decisions.md`, `superpowers/`) и в `/source/prev-project` (бизнес-контекст
сущностей CRM того портала). Из этой истории стоит взять инженерный подход
(TDD с живыми REST-проверками, camelCase у `crm.item.list`, hex-префикс CRM-
привязки для смарт-процессов, батчи вместо foreach), но **не** конкретные
ID/коды полей.

Плейсмент — LEFT_MENU (дефолт, уже настроен в `www/api/bind.php`). Cron не
нужен (`www/bin/process.php` в проекте отсутствует — не переносился).

## Сейчас в работе

Приложение установлено и работает. Открытый хвост (не блокирует
использование): цвета стадий (`color` в `DASHBOARD_*_STAGES`) — всё ещё
плейсхолдеры с воронки первого клиента, не сняты `crm.status.list` с
`ooordi.bitrix24.ru`. Теперь, когда токены есть (install-flow пройден), это
можно сделать в любой момент живым REST-вызовом.
