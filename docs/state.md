## Где мы сейчас

Это копия проекта `ap-pdir`, сделана как стартовая точка для второго клиента
на том же движке (тиражирование, не с нуля с чистого шаблона). Код (`www/`),
инженерные решения и грабли Б24 (`rules/`) перенесены как есть — архитектура
и REST-паттерны те же. Не перенесены: бизнес-данные первого клиента (портал,
ID сущностей/полей, конкретные UF-коды) — они специфичны для того портала и
для нового клиента будут другими.

История первого проекта (для клиента «Альфа», портал `alfa-prj.bitrix24.ru`)
— в [docs/prev-project/](./prev-project/) (`state.md`, `changelog.md`,
`decisions.md`, `superpowers/`) и в `/source/prev-project` (бизнес-контекст
сущностей CRM того портала). Из этой истории стоит взять инженерный подход
(TDD с живыми REST-проверками, camelCase у `crm.item.list`, hex-префикс CRM-
привязки для смарт-процессов, батчи вместо foreach), но **не** конкретные
ID/коды полей — их нужно переснять заново под новый портал.

`.onboarding-done` удалён — онбординг под нового клиента ещё не пройден.

## Сейчас в работе

Ждём от пользователя: (1) актуальную информацию по сущностям Б24 нового
портала в `/source` (аналог `АП Deal/Milestone/Module/Pay/Задачи.md` первого
проекта, только с реальными ID/полями нового клиента), (2) данные для
онбординга (название клиента, домен портала, скоупы, плейсменты), (3) ссылку
на GitHub-репозиторий для этого проекта, (4) данные для регистрации local-app
на новом портале (после чего — `bash deploy.sh` и привязка, слот на VPS rub24
свободен, слаг `rdi-pdir`).

## Следующие шаги

- [ ] Разработчик кладёт актуальные АП-документы по сущностям нового портала в `/source`
- [ ] Пройти онбординг (см. `docs/onboarding.md`) — имя клиента, портал, скоупы, плейсменты
- [ ] Получить ссылку на GitHub-репозиторий, привязать `origin`
- [ ] `bash deploy.sh` на rub24 (slug `rdi-pdir`), завести Caddy-блок `/rdi-pdir`
- [ ] Зарегистрировать local-app на новом портале, вписать `B24_CLIENT_ID`/`SECRET`
- [ ] Пройти install-flow, подтвердить токены
- [ ] Переснять реальные ID/коды полей сущностей (Сделка/Этап/Модуль/Оплата) живыми
      REST-вызовами (`crm.item.fields`) — не полагаться на аналогию с первым проектом
