# Save-tokens-gate: когда можно перезаписывать токены

## Проблема

`www/index.php` получает POST на каждое открытие приложения любым
пользователем портала — не только при установке. Если наивно сохранять
`AUTH_ID`/`REFRESH_ID` из **любого** такого POST, второй сотрудник, открывший
приложение, затрёт токены первого (обычно — админа, который ставил
приложение). Дальше все REST-вызовы начнут ходить под чужим/более слабым
токеном, а если тот сотрудник уйдёт из компании и токен истечёт без
`refresh_token` — приложение сломается необратимо.

## Условие перезаписи (save-policy)

В `www/index.php` / `www/api/install.php`:

```php
$isFirst  = !$b24->hasTokens();
$isFormal = ($post['INSTALL'] ?? '') === 'Y' || ($post['event'] ?? '') === 'ONAPPINSTALL';
if ($isFirst || $isFormal || $b24->isAccessExpired()) {
    $b24->saveTokensFromInstall($post, $referer);
}
```

Перезаписываем только если:
- токенов ещё нет вообще (`$isFirst` — самая первая установка), **или**
- это формальное (пере)событие установки (`INSTALL=Y` / `ONAPPINSTALL`), **или**
- текущий `access_token` истёк (`isAccessExpired()`).

На обычном открытии приложения рядовым пользователем ни одно из условий не
верно → токены не трогаются, админский `access_token`/`refresh_token`
остаётся как есть.
