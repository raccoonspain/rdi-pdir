# crm.item.* вместо legacy crm.<entity>.*

## Правило

Для любых CRM-сущностей предпочитать универсальные методы `crm.item.*`
(`crm.item.get`, `crm.item.list`, `crm.item.add`, `crm.item.update`) с
обязательным параметром `entityTypeId`, а не устаревшие методы под конкретную
сущность (`crm.deal.get`, `crm.lead.get`, `crm.contact.get` и т.д.).

## Почему

- **Смарт-процессы и кастомные сущности** (пользовательские CRM-сущности с
  произвольным `entityTypeId`, например `1036` в `fv-taskindep-panel`) вообще
  **не имеют** legacy-метода — `crm.item.*` единственный способ работать с
  ними через REST.
- Стандартные сущности (сделки, лиды, контакты) имеют и `entityTypeId`
  (2 — сделки, 1 — лиды и т.д.), и legacy-метод — но `crm.item.*` работает
  одинаково для всех, что упрощает код: одна и та же функция читает и сделку,
  и смарт-процесс, разница только в `entityTypeId`.
- Битрикс24 развивает REST API в сторону универсальных методов — legacy
  методы поддерживаются для обратной совместимости, но не получают новых
  возможностей наравне с `crm.item.*`.

## Пример из шаблона

`www/api/zov.php` в `fv-taskindep-panel`:
```php
$b24->batch([
    'zov' => ['crm.item.get', ['entityTypeId' => 1036, 'id' => $id]],
    // ...
]);
```
Смарт-процесс `1036` не имеет своего `crm.<entity>.get` — только `crm.item.get`
с явным `entityTypeId` умеет к нему обратиться.
