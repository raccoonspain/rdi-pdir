# Прямой переход на api/bind.php или api/debug.php падает с «нет сессии»

## Сценарий

Приложение открывается внутри Б24 нормально (сессия работает, доска/UI
грузится). Но стоит перейти по прямой ссылке `https://<host>/<slug>/api/bind.php`
или `.../api/debug.php` из адресной строки того же браузера (как советует
комментарий в `bind.php`: «открой `/api/bind.php` из админ-сессии») — падает
`{"error":"Доступ только из Bitrix24..."}`, хотя сессия только что была
активна.

## Причина

Сессия внутри приложения аутентифицируется двумя способами (см.
`requireSession()` в `api/session.php`): кукой (`gsp_session`) или заголовком
`X-App-Session` (его подставляет `js/app.js`/`js/board.js` в каждый свой
`fetch`). Кука выставляется в ответ на POST, который приходит **внутри
iframe** Б24 — с точки зрения браузера это third-party контекст (домен
приложения ≠ домен портала). Часть браузеров это устройство не пропускает
вообще: Safari блокирует third-party cookies по умолчанию, Chrome
последовательно движется туда же (Privacy Sandbox). `SameSite=None; Secure`
(уже стоит в `setSessionCookie()`) не помогает — блокировка именно
по признаку «третья сторона», а не по `SameSite`.

Итог: кука на сервере реально сохраняется (в `data/sessions.php`), но
браузер её у себя не хранит/не шлёт — обычная прямая навигация на
`api/bind.php` кук не имеет вообще.

## Как диагностировать

Если `placement.bind` из `api/bind.php` (открытого напрямую по ссылке)
падает на `requireSession()`, а внутри самого приложения всё работает —
это почти наверняка это, а не баг в сессии/токенах. Проверить быстро:
открыть DevTools → Application → Cookies на домене приложения — куки
`gsp_session` там не будет, даже если приложение только что открывалось.

## Решение

Не полагаться на прямую навигацию с кукой для служебных эндпоинтов.
Дёргать их тем же способом, что и весь остальной UI — `fetch` с заголовком
`X-App-Session` (значение уже лежит в `window.APP_SESSION`, инжектится
`index.php`):

```js
fetch('api/bind.php', { headers: { 'X-App-Session': window.APP_SESSION || '' } })
  .then(r => r.text())
  .then(html => { document.open(); document.write(html); document.close(); });
```

Практический паттерн — кнопка в топбаре самого приложения (не отдельная
страница), которая это делает и печатает HTML-ответ `bind.php` прямо в
текущий документ. Полезно всегда добавлять такую кнопку в проект, раз
`bind.php` — стандартная часть шаблона и обычно перебиндивается не один раз
за жизнь проекта (смена плейсментов, переустановка карточки под новым
`client_id` — тогда все старые бинды не действуют, см.
[b24-local-app-app-reinstall-application-token-mismatch](b24-local-app-app-reinstall-application-token-mismatch.md)).

`api/debug.php` (REST-тестер) в этом смысле безопаснее — у него есть
собственный GET-интерфейс с формой, и обычно к нему заходят уже находясь
внутри рабочей сессии приложения (тогда кука есть, потому что тот же самый
браузер/вкладка уже открывал(а) приложение через iframe и получил(а) куку
в ответ на его собственный POST). Проблема всплывает конкретно при переходе
**в отдельной вкладке/окне** или после того как кука была заблокирована
браузером с самого начала.
