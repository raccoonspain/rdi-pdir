# Привязка задачи (`UF_CRM_TASK`) к смарт-процессу — префикс `T{hex}`, не буквенный код

## Правило

Общая документация Б24 для UF-поля «Привязка к элементам CRM»
(и для `tasks.task.add` → `UF_CRM_TASK`) описывает формат `{PREFIX}_{ID}`,
где `PREFIX` — краткий буквенный код типа: `L` (Лид), `D` (Сделка),
`C` (Контакт), `CO` (Компания), `SI` (Счёт новый) и т.д. Эта таблица
верна **только для классических CRM-сущностей** (фиксированный
`entityTypeId` — Сделка `2`, Контакт `3`, …).

Для смарт-процессов (Товарные предложения / SPA, `entityTypeId ≥ 1032`,
в частности любые кастомные смарты вроде «Сделка»=1050, «Этап»=1054,
«Модуль»=1062 в этом проекте — см. `/source`) буквенного кода **нет**.
Префикс вычисляется на лету: `entityTypeId` переводится в
шестнадцатеричное представление в **нижнем регистре**, и к нему
приписывается `T`.

```
PREFIX = 'T' . dechex(entityTypeId)
```

Примеры:

| Сущность | `entityTypeId` | `dechex()` | Префикс | Значение для id=123 |
|---|---|---|---|---|
| Классическая Сделка | 2 | `2` | `T2`* | не так — см. ниже |
| Смарт-процесс `entityTypeId=128` (пример из офиц. доки) | 128 | `80` | `T80` | `T80_123` |
| Сделка этого проекта (смарт-процесс) | 1050 | `41a` | `T41a` | `T41a_123` |

\* Это не опечатка-ловушка, а суть правила: классическая Сделка
(`entityTypeId=2`) получает буквенный код `D` (см. таблицу
`entityTypeAbbr` в общей доке), а НЕ `T2` — hex-формула действует
**только** для смарт-процессов. Спутать легко, если судить по названию
объекта в интерфейсе Б24 («Сделка» есть и там, и там) — источник истины
не название, а фактический `entityTypeId` (проверяется через
`crm.item.fields.json?entityTypeId=<N>` — если метод отдаёт `result.fields`
и работает, это `crm.item.*`-сущность, т.е. смарт-процесс).

## Источник

`/home/deploy/refs/b24-rest-docs/api-reference/crm/data-types.md`,
раздел «Формат значений для пользовательского поля „Привязка к
элементам CRM“» — фраза «Для смарт-процессов PREFIX вычисляется из
`entityTypeId`: значение переводится в шестнадцатеричное представление
в нижнем регистре, затем добавляется префикс `T`».

## Как проверить самостоятельно

```php
'T' . dechex($entityTypeId) . '_' . $id
```

Отправить в `tasks.task.add` (`UF_CRM_TASK`) или в любое UF-поле типа
`crm`, привязанное к смарт-процессу, и свериться в карточке созданной
задачи/записи — привязка должна отобразиться корректно, без ошибки
формата.

## Почему это важно

Использование буквенного кода из общей таблицы (например `D_123`)
для смарт-процесса **не даёт ошибки** на вызове `tasks.task.add` — Б24
просто не находит совпадение и молча не привязывает задачу ни к чему
(поле остаётся видимым в карточке, но пустым), либо привязывает не к
той сущности, если `D_123` случайно совпадает с ID другого объекта
классического типа. Не даёт ошибки → легко не заметить при беглой
проверке. См. D-013 и D-011 в `docs/prev-project/decisions.md` (первый
проект на этом движке, `ap-pdir`) — тот же класс граблей: расхождение
общей доки Б24 и реального поведения на конкретном типе поля/сущности,
не полагаться на общую таблицу без проверки конкретного `entityTypeId`.
