# Добавил scope в карточку local-app — уже выданный access_token его не подхватывает сам

## Правило

Добавление нового права (scope) в карточку local-app в Б24-админке и
сохранение карточки **не обновляет ретроактивно** уже выданный
`access_token`. Токен продолжает работать по старому набору прав, любой
вызов метода, требующего нового scope, отдаёт:

```
{"error":"insufficient_scope","error_description":"The request requires
higher privileges than provided by the access token"}
```

даже если в карточке новый scope точно сохранён.

## Фикс — без переустановки приложения

Новый scope подхватывается при следующем обмене `refresh_token` →
`access_token` (`grant_type=refresh_token`, см. `settings/oauth/auto-renewal.md`
в клоне `b24-rest-docs`) — Б24 при таком обмене выдаёт токен с
**текущим** набором прав приложения, не с тем, что было на момент
первичной установки. Достаточно форсировать один такой обмен:

```php
$body = http_build_query([
    'grant_type'    => 'refresh_token',
    'client_id'     => B24_CLIENT_ID,
    'client_secret' => B24_CLIENT_SECRET,
    'refresh_token' => $storedRefreshToken,
]);
// POST на https://oauth.bitrix.info/oauth/token/
// → новый access_token/refresh_token, сохранить через storeWrite(B24_TOKENS_FILE, ...)
```

Штатный код проекта (`B24::refreshAccessToken()` в `www/api/b24.php`)
делает ровно это, но вызывается только **автоматически** при ошибках
`expired_token`/`invalid_token` в `B24::call()` — на `insufficient_scope`
не реагирует (это не «токен истёк», а «токен не той ширины прав»).
Поэтому после добавления scope нужен один осознанный внешний толчок:
либо временный одноразовый скрипт, повторяющий этот вызов (не коммитить,
удалить сразу после использования — см. прецедент в D-013 из первого
проекта на этом движке, `docs/prev-project/decisions.md`), либо (менее
надёжно) переоткрытие приложения в портале, если это по факту triggerит
`ONAPPINSTALL`/повторный install POST на данном портале.

## Почему это важно

Легко принять `insufficient_scope` за «scope не сохранился» и начать
перепроверять карточку приложения по кругу — хотя карточка уже
правильная, просто токен старый. Проверка: в ответе `refresh_token`-обмена
(или в `app.info`) поле `scope` покажет текущий эффективный набор — но
для local-app это поле возвращает буквально строку `"app"`, не список
отдельных прав (гранулярность прав здесь не выражена в этом поле) —
не пытаться распарсить его как список, единственный надёжный тест —
реальный вызов метода, которому нужен новый scope.
