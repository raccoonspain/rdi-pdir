# crm.item.list: `select`/ответ — camelCase, `filter` — оба стиля

## Правило

В `crm.item.*` (универсальные методы, см. [rule-crm-item-universal-api](rule-crm-item-universal-api.md))
параметр `select` понимает **только camelCase**-имена полей, и ответ отдаёт
поля тоже в camelCase — даже если в системе (и в `crm.item.fields`) поле
называется классическим верхним регистром (`UF_CRM_13_O_CODE`, `STAGE_ID`,
`PARENT_ID_1050`).

Правило преобразования — механическое, по `_`:
- первый сегмент **целиком в нижний регистр**
- каждый следующий сегмент — с заглавной буквы, остальное строчными
  (цифры и однобуквенные сегменты не трогаются)
- склеить без разделителей

Примеры (проверено REST-запросом на живом портале):

| Код поля | camelCase |
|---|---|
| `STAGE_ID` | `stageId` |
| `PARENT_ID_1050` | `parentId1050` |
| `UF_CRM_13_O_CODE` | `ufCrm13OCode` |
| `UF_CRM_15_MST_CONTR_PLAN` | `ufCrm15MstContrPlan` |

`filter` **терпимее** — оба варианта (`STAGE_ID` и `stageId`) отработали
одинаково в тесте. Но полагаться на это не стоит для `select` — там
верхний регистр **молча теряет поле** (не ошибка, просто поле отсутствует
в ответе).

## Как проверить самостоятельно

```
crm.item.list?entityTypeId=<N>&select[]=UF_CRM_XX_FIELD   → поля нет в ответе
crm.item.list?entityTypeId=<N>&select[]=ufCrmXxField      → поле есть
```

## Почему это важно

Бизнес-документация проекта (`/source`, `crm.item.fields`) описывает поля
классическими кодами — так их видит админ в интерфейсе Б24 и так они
называются в автоматизациях/роботах. Но код, который читает `crm.item.list`,
должен обращаться к ним camelCase-именами. Если писать `select` копипастой
из `/source` — тихо получишь `null`/отсутствующее поле, а не ошибку, и
это легко не заметить при беглом тестировании (пока не сверишься с полным
ответом без `select`, где Б24 сам возвращает всё в camelCase).

## Практический совет

Не держать в уме два алфавита — писать вспомогательную функцию
`toB24CamelCase(string $code): string`, которая конвертирует по правилу
выше, и гонять код через неё при формировании `select`, а не хардкодить
camelCase-литералы вручную (опечатка в ручной конвертации — тот же тихий
баг).
