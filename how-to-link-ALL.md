# Как привязать приложение к порталу Б24 — произвольный хостинг (1% случаев)

## Зачем этот файл нужен, если есть how-to-link.md

[how-to-link.md](how-to-link.md) описывает **99% случаев** — деплой на наш
собственный VPS (`b24.blackboxbegin.space`), где заранее известно всё:
домен фиксирован, путь всегда `/var/www/b24/<slug>`, веб-сервер — Caddy.
При таком конвенте `APP_URL`/`APP_PATH`/`DATA_ROOT` можно вычислить
заранее из одного значения (имени папки проекта) — именно это и делает
`deploy.sh`, генерируя рабочий `env.php` сам, без единого ручного шага
для этих трёх констант.

Но шаблон `tmpl-b24-php` изначально не привязан к этой конкретной
инфраструктуре — он подходит для **любого хостинга, где работает PHP**:
- **Хостинг заказчика** — сдают доступ по FTP/файловому менеджеру, SSH
  может не быть вообще.
- **Неизвестный путь установки** — заказчик сам решает, в какую
  поддиректорию (или в корень домена) залить файлы; вы это не
  контролируете и не знаете заранее.
- **Неизвестный домен** — у заказчика уже есть свой домен/поддомен,
  который вы видите только по факту.
- **Не Caddy** — может быть Apache, nginx, что угодно, с непредсказуемым
  поведением `DocumentRoot`, реврайтами, иногда с `strip_prefix`-подобной
  логикой в реверс-прокси (та же проблема с автоопределением URL, что и
  у нас, только не факт, что вы вообще узнаете об этом заранее).

В этом сценарии `deploy.sh` **не работает** — он жёстко заточен под
наш VPS (`rsync` в `/var/www/b24/<slug>`, фиксированный домен). Взамен
шаблон везёт с собой `www/init.php` — скрипт, который **сам определяет
хост-пути по факту HTTP-запроса**, в котором его открыли, независимо от
того, куда и как физически залили файлы.

Это и есть причина, по которой `init.php` остаётся в шаблоне, хотя в
основном (VPS) flow он не используется: без гарантии на конвент
инфраструктуры автоопределение через `init.php` — единственный
универсальный способ узнать `APP_URL`/`APP_PATH`.

---

## Что такое client_id и client_secret

Битрикс24 использует OAuth-авторизацию. Когда пользователь открывает
приложение, Б24 выдаёт временный код, который приложение обменивает на
`access_token` для REST-запросов. Для этого обмена нужны:

| Параметр | Назначение |
|---|---|
| `client_id` | Идентификатор приложения — говорит Б24, *какое* приложение делает запрос |
| `client_secret` | Секретный ключ — доказывает, что запрос делает *настоящий* владелец приложения |

Хранятся в `env.php` рядом с приложением. **Никогда не коммитить в git.**

## Порядок действий

### 1. Залить файлы на хостинг

Скопировать содержимое папки `www/` (**включая `init.php`** — в отличие
от VPS-флоу, здесь он нужен) на хостинг любым доступным способом: FTP,
файловый менеджер хостинг-панели, `scp`/`rsync` по SSH, если он есть.

Путь назначения — на усмотрение заказчика/хостинга: может быть корень
домена, может быть поддиректория. `init.php` разберётся с этим сам на
следующем шаге.

### 2. Открыть init.php в браузере

```
https://<фактический-домен>/<фактический-путь>/init.php
```

Скрипт:
- создаст `env.php` рядом с собой,
- определит `APP_URL`/`APP_PATH`/`DATA_ROOT` по данным реального
  запроса (`HTTP_HOST`, `SCRIPT_NAME`),
- покажет на странице готовые значения для карточки local-app (handler
  URL, install URL, скоупы) с кнопками «Копировать».

### 3. Проверить APP_URL на странице

Страница `init.php` показывает, каким получился `APP_URL`. Сверьте его
с тем адресом, по которому вы реально открыли страницу в браузере.

**Если хостинг стоит за реверс-прокси, который срезает префикс пути**
(поведение похоже на наш `uri strip_prefix` в Caddy) — `APP_URL` придёт
без этого префикса. Страница сама покажет предупреждение в этом
случае. Тогда правим вручную в `env.php`:

```php
// Было (без префикса):
define('APP_URL', 'https://клиент-домен.ру');

// Должно быть (с реальным путём установки):
define('APP_URL', 'https://клиент-домен.ру/фактический-путь');
```

Если реверс-прокси нет (большинство обычных shared-хостингов) —
`APP_URL` окажется верным сразу, править не нужно.

### 4. Зарегистрировать приложение в Б24

В портале Битрикс24: **Приложения → Разработчикам → вкладка «Готовые
сценарии» → Другое → Локальное приложение**

- Тип: **Серверное**
- **Путь обработчика:** значение `handler URL` со страницы `init.php`
  (оканчивается на `/index.php`)
- **Путь установки:** тот же URL
- Права (минимум): `crm`, `user`, `placement`

После сохранения Б24 покажет `client_id` и `client_secret`.

### 5. Вписать credentials в env.php

Через файловый менеджер хостинга (или SSH, если есть) открыть `env.php`
и вписать:

```php
define('B24_CLIENT_ID', 'app.xxxxxxxxx.xxxxxxxx');       // из карточки Б24
define('B24_CLIENT_SECRET', 'xxxxxxxxxxxxxxxxxxxxxxxx');  // из карточки Б24
```

### 6. Права на data/

На нашем VPS `deploy.sh` сам выставляет `chmod 0777` на `data/`, потому
что известно, что PHP-FPM работает от `www-data`, а файлы заливаются от
`deploy`. На чужом хостинге модель пользователей может быть другой
(часто PHP и FTP-аккаунт работают от одного и того же юзера — тогда
права по умолчанию уже подойдут). Если приложение падает с ошибкой
записи в `data/` — через файловый менеджер хостинга выставить права
`0755` или, если не помогло, `0777` на папку `data/`.

Защита содержимого — не права ФС, а `<?php exit;?>`-префикс в самих
файлах стора, так что широкие права на папку не проблема безопасности.

### 7. Удалить init.php

Разово отработал — снимите его с хостинга через файловый менеджер, как
и предупреждает сама страница `init.php` при выводе.

### 8. Установить приложение

Открыть приложение из меню Б24 или перейти по `index.php` напрямую.
Двухфазный install-flow отработает сам, OAuth-токены сохранятся в
`data/b24-tokens.php`.

## Когда возвращаться к обычному how-to-link.md

Если в процессе выяснилось, что хостинг заказчика на самом деле — это
наш собственный VPS (`b24.blackboxbegin.space`) или другой сервер с тем
же предсказуемым конвентом (домен+путь известны заранее) — используйте
[how-to-link.md](how-to-link.md) и `deploy.sh`, они проще и не требуют
ручной правки `APP_URL`.
